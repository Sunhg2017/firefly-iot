# 协议解析规则分步抽屉使用说明

## 1. 适用范围

本文档用于说明“协议解析规则”页面中新建、编辑抽屉改为分步式配置后的使用方式，适用于租户侧规则维护人员、实施人员和联调人员。

## 1.1 页面浏览方式

协议解析页面现在分为两个主视图：

- `规则维护`
- `运行时状态`

建议按职责使用：

- 维护规则、筛选规则、查看规则列表时进入“规则维护”
- 查看运行时指标、插件列表、插件目录、执行插件重载时进入“运行时状态”

## 2. 页面变化

- 协议解析规则的新建和编辑不再使用一整张超长表单。
- 抽屉统一拆分为 5 个步骤：
  1. 模板与作用域
  2. 协议与匹配
  3. 解析实现
  4. 发布策略
  5. 预览确认
- 页面顶部新增总览区，用于快速确认当前租户、当前产品筛选、规则总数和插件装载数量。

## 3. 推荐操作流程

### 3.1 模板与作用域

在第一步先完成两件事：

- 选择最接近业务场景的模板
- 选择规则作用域

说明：

- 产品级规则需要选择具体产品。
- 租户默认级规则不需要选择产品。
- 模板应用后会覆盖解析相关字段，但会保留当前已经选择的作用域和产品。

### 3.2 协议与匹配

第二步完成规则的入口条件：

- 协议
- 传输方式
- 方向
- 超时时间
- 解析方式
- 拆帧模式
- 异常策略
- 匹配规则 JSON
- 拆帧配置 JSON

建议：

- 优先使用页面提供的 JSON 预设按钮，而不是从零手写。
- TCP/UDP 场景要特别注意“协议”和“传输方式”不是同一概念。

### 3.3 解析实现

第三步完成实际解析逻辑：

- 维护 `parserConfigJson`
- 维护 `visualConfigJson`
- 选择脚本模式或插件模式

脚本模式：

- 维护脚本语言
- 维护 `scriptContent`

插件模式：

- 直接从下拉中选择 `pluginId`
- 如有可选版本，再选择 `pluginVersion`

### 3.4 发布策略

第四步配置规则发布范围：

- 全量发布
- 设备名单
- 百分比灰度

说明：

- 页面继续提供灰度配置预设。
- 新配置优先使用 `deviceNames`。

### 3.5 预览确认

最后一步会集中展示：

- 关键摘要
- 匹配规则预览
- 拆帧配置预览
- 解析配置预览
- 可视化流预览
- 脚本或插件信息
- 发布配置预览

保存前建议在这里统一复核一次。

## 4. 交互规则

- 点击“下一步”时，只校验当前步骤需要填写的字段。
- 可以返回上一步继续修改。
- 抽屉会保留之前已经填写的内容，不会因为切步骤而丢失。
- 只有到最后一步才执行最终保存。

## 4.1 编辑器能力

协议解析规则里的关键 JSON 和脚本字段已经升级为代码编辑器：

- `matchRuleJson`
- `frameConfigJson`
- `parserConfigJson`
- `visualConfigJson`
- `scriptContent`
- `releaseConfigJson`

现在可以直接获得：

- JSON / JavaScript 语法高亮
- JSON 结构提示和字段补全
- `parse(ctx)`、`encode(ctx)` 脚本片段补全
- `ctx.topic`、`ctx.payloadText`、`ctx.config` 等上下文字段提示

使用建议：

- JSON 字段优先配合自动提示录入关键属性，再按业务需要补充细节。
- 脚本模式优先通过 `parse(ctx)` 或 `encode(ctx)` 片段起步，再做局部修改。
- 如果只是微调结构，直接在编辑器里操作会比纯文本框更容易发现括号、逗号和字段名问题。

## 5. 常见问题

### 5.1 为什么不能直接跳到后面的步骤？

新建规则时建议按顺序配置，避免后面的解析实现依赖前面的协议、方向、产品作用域却还未确定。

### 5.2 为什么最后增加了预览确认？

协议解析规则通常包含多段 JSON、脚本或插件配置。增加最终确认步骤后，可以在保存前统一发现误配问题，减少反复进入抽屉修改。

### 5.3 返回前一步后，之前的内容还在吗？

会保留。当前实现已经去掉会导致字段卸载丢失的表单配置，步骤切换不会清空已填写内容。
