# 协议解析规则分步抽屉使用说明

## 1. 适用范围

本文档用于说明“协议解析规则”页面中新建、编辑抽屉改为分步式配置后的使用方式，适用于租户侧规则维护人员、实施人员和联调人员。

## 1.1 页面浏览方式

协议解析页面现在分为两个主视图：

- `规则维护`
- `运行时状态`

页面顶部会一直显示“页面总览”，用于确认当前租户、产品筛选上下文、规则总数和已加载插件数。

建议按职责使用：

- 维护规则、筛选规则、查看规则列表时进入“规则维护”
- 查看运行时指标、插件列表、插件目录、执行插件重载时进入“运行时状态”

## 2. 页面变化

- 协议解析规则的新建和编辑不再使用一整张超长表单。
- 抽屉统一拆分为 5 个步骤：
  1. 模板与作用域
  2. 协议与匹配
  3. 解析实现
  4. 发布策略
  5. 预览确认
- 页面顶部新增总览区，用于快速确认当前租户、当前产品筛选、规则总数和插件装载数量。
- 只有在“规则维护”标签页中才会显示“筛选条件”卡片；运行时相关内容统一收口到“运行时状态”。
- 首次进入 JSON 或脚本编辑步骤时，编辑器会按需加载对应资源；首次稍慢属于正常现象，后续再次打开会更快。
- 编辑抽屉右侧新增固定侧栏，统一显示步骤进度、当前步骤检查和关键摘要。
- 抽屉与全屏编辑现在只保留当前操作直接相关的提示；模板背景、字段解释和联动规则统一查看本说明文档。
- 抽屉宽度已扩大，JSON 与脚本编辑时可以同时看到表单和摘要，减少来回切步骤确认。
- 新建规则时，系统会先按当前协议场景带出推荐的匹配规则和解析配置，不需要从空白 JSON 开始。

## 3. 推荐操作流程

### 3.1 模板与作用域

在第一步先完成两件事：

- 选择最接近业务场景的模板
- 选择规则作用域

说明：

- 产品级规则需要选择具体产品。
- 租户默认级规则不需要选择产品。
- 模板应用后会覆盖解析相关字段，但会保留当前已经选择的作用域和产品。

### 3.2 协议与匹配

第二步完成规则的入口条件：

- 协议
- 传输方式
- 方向
- 超时时间
- 解析方式
- 拆帧模式
- 异常策略
- 匹配规则 JSON
- 拆帧配置 JSON

建议：

- 优先使用页面提供的 JSON 预设按钮，而不是从零手写。
- TCP/UDP 场景要特别注意“协议”和“传输方式”不是同一概念。
- 录完一段 JSON 后，可以直接在右侧点“格式化 JSON”，先把结构整理好再继续填。
- 如果你在这里改了协议、方向或拆帧模式，而对应 JSON 还没手改过，页面会自动切到新的推荐默认值。
- 如果匹配规则或拆帧配置已经比较长，可以直接点字段旁边的“全屏编辑”。

### 3.3 解析实现

第三步完成实际解析逻辑：

- 维护 `parserConfigJson`
- 维护 `visualConfigJson`
- 选择脚本模式或插件模式
- `parserConfigJson` 和 `visualConfigJson` 也支持全屏编辑，适合长配置连续调整。

脚本模式：

- 维护脚本语言
- 维护 `scriptContent`
- 脚本编辑器会提供 `parse(ctx)`、`encode(ctx)` 等片段补全，以及 `ctx` 相关上下文提示。
- 如果脚本较长，可直接点“全屏编辑”，在大窗口里连续修改。
- 如果想快速起步，可直接点“当前方向骨架”；如果已经维护了可视化配置，可点“按可视化重建”。

插件模式：

- 直接从下拉中选择 `pluginId`
- 如有可选版本，再选择 `pluginVersion`

### 3.4 发布策略

第四步配置规则发布范围：

- 全量发布
- 设备名单
- 百分比灰度

说明：

- 页面继续提供灰度配置预设。
- 新配置优先使用 `deviceNames`。
- 如果灰度配置还是系统默认值，切换发布方式后会自动跟随成对应示例。
- `releaseConfigJson` 也支持全屏编辑，适合维护较长的设备名单。

### 3.5 预览确认

最后一步会集中展示：

- 关键摘要
- 匹配规则预览
- 拆帧配置预览
- 解析配置预览
- 可视化流预览
- 脚本或插件信息
- 发布配置预览

保存前建议在这里统一复核一次。

如果预览页右侧仍提示“还有待补齐项”，说明前面的某一步已经被改成不合法，需要按右侧提示返回修复。
预览卡片右上角现在也能直接“回到本步”或打开对应字段的“全屏编辑”。

## 4. 交互规则

- 点击“下一步”时，只校验当前步骤需要填写的字段。
- 可以返回上一步继续修改。
- 抽屉会保留之前已经填写的内容，不会因为切步骤而丢失。
- 只有到最后一步才执行最终保存。
- 右侧步骤卡片可以直接返回已到达的步骤，适合从摘要里发现问题后快速回改。

## 4.1 右侧侧栏怎么用

右侧固定侧栏建议这样使用：

- 先看“编辑进度”，确认前四步是否都已经就绪。
- 再看“当前步骤检查”，这里会直接列出还缺什么字段或哪段 JSON 不合法。
- 如果当前步骤包含 JSON，优先点“格式化 JSON”，再点“校验本步”。
- “当前摘要”会持续展示作用域、协议/传输、解析方式、发布方式等关键信息，适合边填边核对。
- 当你切换协议、方向、拆帧或发布方式后，也可以先在这里确认默认 JSON 是否已经自动同步。
- 如果某段 JSON 已经很长，优先进入字段自己的“全屏编辑”，再在窗口内格式化。
- 如果是在预览确认页里发现问题，直接点摘要里的“回到第 N 步”会更快。
- 如果 JSON 被改乱了，全屏窗口里的“恢复推荐值”可以直接回到当前上下文下的推荐配置。

## 4.2 编辑器能力

协议解析规则里的关键 JSON 和脚本字段已经升级为代码编辑器：

- `matchRuleJson`
- `frameConfigJson`
- `parserConfigJson`
- `visualConfigJson`
- `scriptContent`
- `releaseConfigJson`

现在可以直接获得：

- JSON / JavaScript 语法高亮
- JSON 结构提示和字段补全
- `parse(ctx)`、`encode(ctx)` 脚本片段补全
- `ctx.topic`、`ctx.payloadText`、`ctx.config` 等上下文字段提示

使用建议：

- JSON 字段优先配合自动提示录入关键属性，再按业务需要补充细节。
- 脚本模式优先通过 `parse(ctx)` 或 `encode(ctx)` 片段起步，再做局部修改。
- 如果只是微调结构，直接在编辑器里操作会比纯文本框更容易发现括号、逗号和字段名问题。
- JSON 输入报错时优先看右侧“当前步骤检查”，通常能比最终保存失败更早定位问题。
- 全屏编辑和抽屉里的脚本编辑器共用同一份内容，关闭全屏后不需要再粘贴回去。

## 5. 常见问题

### 5.1 为什么不能直接跳到后面的步骤？

新建规则时建议按顺序配置，避免后面的解析实现依赖前面的协议、方向、产品作用域却还未确定。

### 5.2 为什么最后增加了预览确认？

协议解析规则通常包含多段 JSON、脚本或插件配置。增加最终确认步骤后，可以在保存前统一发现误配问题，减少反复进入抽屉修改。

### 5.3 返回前一步后，之前的内容还在吗？

会保留。当前实现已经去掉会导致字段卸载丢失的表单配置，步骤切换不会清空已填写内容。

### 5.4 为什么预览确认页还会提示前面步骤没完成？

因为预览页现在会继续根据前四步的实时内容做检查。如果你在前面改动过 JSON、脚本或插件字段导致内容失效，预览页右侧会直接提示需要回去修哪一步，避免点保存后才发现失败。

### 5.5 为什么有时候切换协议后 JSON 会自动变化，有时候又不会？

因为页面只会自动同步“还保持系统默认值”的那部分 JSON，帮助你少点几次预设按钮；一旦你已经手工改过该字段，系统就会尊重你的修改，不再自动覆盖。

### 5.6 全屏编辑里的脚本会不会和抽屉里的内容不一致？

不会。全屏编辑和抽屉里的脚本字段绑定的是同一个表单值，你在任意一边修改，另一边都会保留同一份最新内容。

### 5.7 JSON 全屏编辑和右侧“格式化 JSON”有什么区别？

右侧“格式化 JSON”是按当前步骤批量处理；字段自己的全屏编辑更适合单独修改一大段 JSON，并且可以只格式化当前这一个字段。

### 5.8 预览确认页里看到问题后，最快怎么改？

优先点对应卡片上的“回到本步”；如果是长 JSON 或长脚本，直接点“全屏编辑”会更省切换成本。

### 5.9 “当前方向骨架”“按可视化重建”“恢复推荐值”分别什么时候用？

- `当前方向骨架`：脚本还没写或想从当前上行/下行方向重新起步时用。
- `按可视化重建`：你已经维护了 `visualConfigJson`，想让脚本重新跟随它时用。
- `恢复推荐值`：适用于 JSON 全屏编辑，快速回到当前协议、方向、拆帧或发布方式下的推荐默认值。
