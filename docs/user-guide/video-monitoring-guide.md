# 视频设备使用说明

## 1. 入口

- 从租户空间进入 `设备资产 -> 设备管理`
- 切到 `视频设备` 视图
- 或从摄像头产品页点击 `视频接入` 直接跳转进入

## 2. 从产品页进入时会发生什么

1. 页面自动切到视频设备视图。
2. 自动打开新增视频设备抽屉。
3. 自动带入当前产品。
4. 自动锁定与产品一致的协议。

如果你不是从产品页进入，也可以手动在抽屉里选择摄像头产品。

## 3. 新建视频设备

1. 点击 `新增视频设备`
2. 选择摄像头产品
3. 填写设备名称
4. 按协议填写接入参数
5. 点击保存

### 3.1 GB28181

必填项：

- `GB 设备编号`
- `GB 域`
- `传输方式`
- `启用认证`（默认开启）
- `认证用户名`（默认带出 `GB 设备编号`，你也可以手工改成其他用户名）
- `认证密码`（开启认证时必填）

播放说明：

- 点击播放后，平台会先向 ZLMediaKit 打开专用 RTP 收流口，再向设备发送 INVITE。
- 如果该设备的 `rtp/{streamId}` 已经在平台内活跃，再次点击播放会直接复用当前会话，不会重复开 RTP 端口。
- 如媒体服务返回的 RTP 端口字段位置或类型存在差异，平台会自动兼容顶层 `port`、`data.port` 以及字符串端口。
- 若平台日志提示 `ZLMediaKit API` 返回 `404`，通常表示媒体服务地址配错到了网关或其他 HTTP 服务，需要运维修正 `zlmediakit.api-host/api-port`。
- 当前开发环境默认连接 `192.168.123.102:18080` 的 ZLM；若你本地没有覆盖配置，但日志仍显示连接 `localhost:18080`，说明服务还未使用最新配置启动。
- 若日志出现 `code=-100, msg=Please login first`，说明当前节点上的 ZLM 很可能还是旧的 `master` 容器，需让运维按最新 compose 重建到仓库内置的稳定版 `firefly-zlmediakit:8.0-588d9de`。
- 若日志出现 `创建rtp端口 :::10000 失败:address already in use`，说明运维仍在使用错误的 ZLM `rtp_proxy.port` 配置，需要把该值改回 `0` 后重启 ZLM。
- 若平台通过 compose 部署，运维需先确认 `zlmediakit` 基础设施已启动，且 `ZLM_RTP_PORT` 与媒体服务配置一致。
- GB28181 流在平台内统一挂到 `rtp/{streamId}`，因此网页播放、截图、录像都基于这一路流。

### 3.2 RTSP / RTMP

必填项：

- `视频源地址`
- `认证用户名 / 认证密码`（只有上游源要求认证时才需要填写；本地摄像头模式固定要求填写）

建议直接填写无凭据地址，例如：

- `rtsp://192.168.1.10:554/live/cam01`
- `rtmp://192.168.1.10/live/cam01`

注意：

- `sourceUrl` 不能写成 `rtsp://user:pass@...` 或 `rtmp://user:pass@...`，旧地址需要人工拆出后再保存。
- 外部源若启用认证，平台只会在运行时把用户名密码拼到上游拉流地址，不会把凭据写回设备资产。
- 本地摄像头模式会把平台资产中的 `sourceUrl` 保持为无凭据地址，真正给 FFmpeg 和平台回拉使用的运行时地址会自动追加 `authUser/authPass`。

播放说明：

- 点击播放后，平台会让 ZLMediaKit 拉取对应源地址。
- 如果该设备的 `live/{streamId}` 已经在平台内活跃，再次点击播放会直接复用当前会话，不会再次创建代理流。
- 如果 ZLMediaKit 里残留了旧的代理任务，平台会先自动清理陈旧代理，再重新拉流；正常情况下不再需要手工进 ZLM 删除代理。
- 网页播放、截图、录像都基于这一路 `live/{streamId}` 代理流。
- 如果该设备来自模拟器的 `本地摄像头` 模式，列表里的 IP 列显示的是模拟器主机地址；`sourceUrl` 仍只展示不带凭据的 ZLM 地址。

### 3.3 编辑时的认证规则

- 编辑设备时，如果你不想改密码，可以把 `认证密码` 留空，平台会保留原密码。
- 如果你关闭 `启用认证`，平台会同步清空已保存的认证用户名和认证密码。
- 对 `RTSP / RTMP + 本地摄像头`，必须保持认证开启，否则保存会直接失败。

## 4. 列表与查询

- 首次进入页面会自动查询一次
- 修改筛选条件后不会自动刷新
- 需要点击 `查询` 按钮才会重新加载

## 5. 通道与媒体控制

在视频设备列表中可以执行：

- 查看通道
- 播放
- 停止
- 截图
- 目录查询
- 设备信息查询
- 云台控制
- 开始录像
- 停止录像

所有媒体控制都按平台 `deviceId` 调用。

## 6. 常见问题

### 6.1 为什么同一台设备不能重复创建

平台会按接入标识做唯一校验：

- `GB28181` 用 `gbDeviceId`
- `RTSP / RTMP` 用完整 `sourceUrl`

### 6.2 为什么保存成功后列表里没有

先检查：

- 设备资产是否创建成功
- 当前账号是否有该设备的数据权限
- 是否误切到了普通设备视图

### 6.3 为什么认证信息正确但还是失败

先检查：

- `GB28181` 场景下，设备端是否使用页面里的 `认证用户名` 参与 Digest；如果你没有改过，一般应与 `GB 设备编号` 一致
- `GB28181` 场景下，`gbDeviceId` 和 `gbDomain` 是否与设备端一致
- `RTSP / RTMP` 场景下，`sourceUrl` 是否仍然误写成了内嵌用户名密码的旧地址
- `RTSP / RTMP + 本地摄像头` 场景下，设备是否保持启用认证，且用户名密码已补齐

### 6.4 为什么页面里找不到旧的视频监控菜单

因为视频设备已经并入 `设备管理`，不再保留独立 `视频监控` 页面和菜单。

### 6.5 播放提示网络错误怎么办

先检查：

- 平台返回的播放地址是否还是 `localhost`
- 运维是否已把 `zlmediakit.public-host/public-port/public-scheme` 配成浏览器可访问地址
- 设备是否在线并且已成功开流

如果是同一台 `GB28181` 设备关闭播放器后再次播放，平台会优先复用现有流；若仍提示启动失败并看到 `This stream already exists`，说明媒体服务还未发布包含重播修复的版本。

如果是同一台 `RTSP / RTMP` 设备再次播放时提示 `This stream already exists`，平台正常行为应直接复用现有 `live/{streamId}` 会话，或者自动清理陈旧代理后再重建；若仍失败，说明媒体服务还未发布包含陈旧代理清理修复的版本。

### 6.6 云台控制提示参数错误怎么办

先检查：

- 当前设备是否为 `GB28181`
- `command` 是否为合法 PTZ 指令（数字或枚举名）
- 速度参数是否为合理范围
