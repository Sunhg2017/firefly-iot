# 设备消息路由补全使用说明
> 适用角色: 运维、联调开发、设备接入开发
> 日期: 2026-03-14

## 1. 这次补全后有什么变化

设备侧以下几类消息现在都会被平台真正处理，而不再只是简单打日志：

- 上下线消息
- 服务调用回执
- 属性设置回执
- OTA 进度

## 2. 你可以看到的直接变化

- 设备上下线后，设备管理页在线状态会及时变化
- 属性设置成功后，设备影子里的 `desired` 会按确认结果收敛
- OTA 进度会进入设备事件列表
- 服务调用回执也会进入设备事件列表
- 如果规则引擎配置了对应规则，这些消息也可以继续参与规则判断

## 3. 联调时怎么判断正常

### 3.1 上下线

- 发 `DEVICE_ONLINE`
  - 设备状态应变为在线
  - 事件列表应出现 `DEVICE_ONLINE`
- 发 `DEVICE_OFFLINE`
  - 设备状态应变为离线
  - 事件列表应出现 `DEVICE_OFFLINE`

### 3.2 属性设置回执

- 云端先设置影子 `desired`
- 设备回传 `PROPERTY_SET_REPLY`
- 平台应看到：
  - `reported` 更新
  - 已确认的 `desired` 键被移除
  - 事件列表新增一条 `PROPERTY_SET_REPLY`

### 3.3 OTA 进度

- 设备回传 `OTA_PROGRESS`
- 平台应看到设备事件列表新增对应进度记录

### 3.4 服务调用回执

- 设备回传 `SERVICE_REPLY`
- 平台应看到设备事件列表新增对应回执记录

## 4. 注意事项

- `PROPERTY_SET_REPLY` 最好返回设备实际确认后的属性值，而不是空 payload
- 生命周期和回执消息现在也会继续转发规则引擎，如果配置了规则，可能会触发通知或动作
- 如果设备消息里 `tenantId`、`deviceId` 不准确，平台会跳过归属不合法的数据处理
