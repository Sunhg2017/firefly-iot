# 审计日志接口可用性修复运维说明

## 1. 适用范围

本文档用于处理系统运维空间 `审计日志` 页面进入后提示“加载审计日志失败”的问题。

涉及模块：

- `firefly-system`

## 2. 变更内容

- 删除审计日志 Controller、Service、AOP 记录器上的条件开关
- 删除 `application.yml` 中默认关闭审计能力的 `firefly.audit.enabled: false`

修复后：

- `/api/v1/audit-logs/list` 会随系统服务正常注册
- `/api/v1/audit-logs/{id}` 会随系统服务正常注册
- `@Auditable` 标记的接口会继续写入审计日志

## 3. 部署说明

发布 `firefly-system` 即可，不需要同步发布前端。

原因：

- 前端页面和请求路径本身已正确存在
- 根因在后端接口未注册

## 4. 数据与 SQL

- 本次不涉及新增、删除或修改 Flyway SQL
- 本次不涉及菜单、权限点、图标或所属空间调整
- 依赖既有 `audit_logs` 表存在

## 5. 验证步骤

1. 发布新的 `firefly-system`
2. 登录系统运维空间
3. 进入 `安全审计 -> 审计日志`
4. 确认页面不再弹出“加载审计日志失败”
5. 调用带 `@Auditable` 的管理接口后重新打开页面，确认能看到最新审计记录
6. 随机点开一条详情，确认详情接口可正常返回

## 6. 排障

### 6.1 页面仍提示加载失败

优先检查：

- 当前运行的 `firefly-system` 是否已升级到本次版本
- 网关路由是否已把 `/api/v1/audit-logs/**` 转发到 `firefly-system`
- 当前账号是否具备 `audit:read` 权限

### 6.2 页面能打开但没有新记录

优先检查：

- 目标接口是否带有 `@Auditable`
- `audit_logs` 表是否存在
- 数据库写入权限是否正常
- 系统日志中是否出现 `Failed to record audit log`

## 7. 回滚说明

- 如需回滚，只需回滚 `firefly-system`
- 本次未改数据库结构，无需执行数据库回滚
- 回滚后页面会再次回到“菜单存在但接口不可用”的旧状态，应谨慎操作
