# 前端表单容器运维说明

## 1. 变更说明

本次发布新增一条前端交互强制规则：

- 复杂表单使用抽屉。
- 抽屉表单如果已经明显过长，必须按业务模块拆分成步骤式配置。

同步落地页面：

- 告警规则配置抽屉

## 2. 发布影响

### 2.1 影响范围

- `firefly-web` 前端页面交互
- 告警规则的新建与编辑流程

### 2.2 不影响范围

- 后端接口协议
- 数据库存储结构
- 告警规则表达式结构

## 3. 发布前检查

执行前端构建：

```bash
cd firefly-web
npm run build
```

重点检查以下内容：

1. 告警规则抽屉顶部应显示步骤条，而不是单页长表单。
2. “上一步 / 下一步”操作后，已填写内容不能丢失。
3. 已到达过的步骤应允许点击回退，未到达步骤不应允许直接跳转。
4. “下一步”应只校验当前模块，不应被后续模块阻塞。
5. 如果选择了通知方式但未选择接收组或指定接收人，应给出校验提示。
6. 最后一步提交后，应继续沿用原有保存成功、关闭抽屉、刷新列表逻辑。

## 4. 发布后验收

### 4.1 页面行为

- 新建告警规则时默认从“基本信息”开始。
- 编辑告警规则时默认从第一步开始，但原有规则数据应正确回填。
- 回退到已配置步骤后再次前进，不应清空已填内容。

### 4.2 交互一致性

- 简单操作继续使用弹窗。
- 复杂配置继续使用抽屉。
- 长抽屉配置应优先改造成步骤式抽屉。

### 4.3 编码检查

- 相关页面和文档中的中文文案必须正常显示。
- 如发现乱码，优先检查文件编码是否被误保存为非 UTF-8。

## 5. 常见问题排查

### 5.1 抽屉仍然显示为单页长表单

排查顺序：

1. 确认前端静态资源已更新到最新版本。
2. 清理浏览器缓存后重新访问。
3. 检查当前部署版本是否包含本次前端提交。

### 5.2 步骤切换后内容丢失

排查顺序：

1. 检查是否复用了同一个 `Form` 实例。
2. 检查关闭抽屉时是否误触发了整表重置。
3. 检查步骤切换是否使用了重新挂载整张表单的实现方式。

### 5.3 下一步无法进入

排查顺序：

1. 检查当前步骤是否存在必填项未完成。
2. 检查规则等级块动态字段是否生成了正确校验路径。
3. 检查通知方式是否已选择但未配置任何接收对象。

## 6. 回滚说明

本次仅涉及前端交互和文档，不涉及后端协议与数据库变更。如需回滚，回滚 `firefly-web` 对应版本并同步回滚这组三类文档，避免页面行为与文档说明不一致。
