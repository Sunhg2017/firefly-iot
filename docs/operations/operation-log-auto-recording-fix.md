# 操作日志自动记录修复运维说明

## 1. 适用范围

本文档用于处理“操作日志页面能打开，但始终没有任何记录”的问题。

涉及模块：

- `firefly-system`

## 2. 变更内容

- `OperLogAspect` 不再只依赖 `@OperLog`
- 系统控制器请求默认自动写入 `operation_logs`
- 日志模块、描述和操作类型从现有 `@Tag`、`@Operation` 与请求信息自动推导
- 跳过 `OperationLogController`、`AuditLogController`、`LoginLogController` 以及 `@Hidden` 内部接口
- 新增 `OperLogAspectTest` 覆盖操作类型归类和跳过规则

## 3. 部署说明

只需发布 `firefly-system`。

原因：

- 前端查询接口与页面原本已可用
- 根因在后端没有任何写入入口

## 4. 数据与 SQL

- 本次不涉及新增或变更 Flyway SQL
- 继续复用既有 `operation_logs` 表
- 本次不涉及菜单、权限点、图标或路由调整

## 5. 验证步骤

1. 发布新的 `firefly-system`
2. 以系统运维管理员登录后台
3. 执行至少一个系统管理动作，例如：
   - 新建或更新用户
   - 修改系统配置
   - 修改租户状态
   - 创建或更新 AppKey
4. 进入 `安全审计 -> 操作日志`
5. 确认能看到刚才的操作记录
6. 点开详情，确认请求路径、请求方法、状态、耗时等字段有值

## 6. 排障

### 6.1 仍然没有记录

优先检查：

- 当前运行的 `firefly-system` 是否已升级到本次版本
- 是否实际触发了 `firefly-system` 的控制器接口，而不是只操作了其他微服务
- 数据库中的 `operation_logs` 表是否存在且可写

### 6.2 页面只有查询日志或日志噪音过多

本次已经跳过安全日志相关查询控制器；如果后续仍出现噪音，应继续检查是否有新的日志查询接口未纳入跳过范围。

### 6.3 操作类型显示不准确

优先检查：

- 对应控制器方法上的 `@Operation(summary = ...)` 是否语义明确
- 是否需要为极少数特殊接口补 `@OperLog` 做显式覆盖

## 7. 回滚说明

- 如需回滚，只需回滚 `firefly-system`
- 本次未改数据库结构，无需数据库回滚
- 回滚后会恢复到“有操作日志页面，但默认没有写入来源”的旧状态
