# 设备模拟器运维说明

## 1. 构建

```bash
cd firefly-simulator
npm run build:vite
```

说明：

- 模拟器通过 `@ffmpeg-installer/ffmpeg` 内置本地推流运行时，不需要额外手工安装 ffmpeg。

## 2. 发布前检查

1. 当前环境登录、环境切换、登录态缓存正常。
2. Video 模式 IPC 已拆分为 `DEVICE` 资产同步和 `MEDIA` 控制两组。
3. 本地状态字段已统一为 `platformDeviceId`。
4. Video 顶部状态卡口径正确：
   - `GB28181` 显示 `SIP 注册`、`SIP 心跳`
   - `RTSP / RTMP` 显示 `平台同步`、`连接状态`
5. 左侧设备列表可直接编辑已有设备，保存后配置能持久化。
6. 编辑在线设备时会先断开旧连接，再由用户重新连接验证新配置。
7. Video 详情页的参数弹窗可直接修改本地摄像头和采集模式，不再只能沿用默认 `15fps`。
8. macOS 本地摄像头起流时，若 `avfoundation` 拒绝默认输入像素格式，系统会继续按 `Supported pixel formats` 自动重试。
9. 环境管理里已补齐 `ZLM 推流主机 / RTSP 端口 / RTMP 端口`，默认值与共享 ZLM `192.168.123.102:18554/1935` 对齐。
10. Video 设备配置已统一改成 `authEnabled/authUsername/authPassword`；历史本地配置中的 `sipPassword` 需人工重新录入到统一认证字段。

## 3. 联调检查

### 3.1 GB28181

1. 点击连接后会先同步设备资产，再自动启动 SIP、发送 REGISTER，并在注册成功后自动开始心跳。
2. 若已启用认证，`认证用户名 / 认证密码` 必须已配置；未手改用户名时默认使用 `GB 设备编号`。
3. 如有多路本机摄像头，先在设备高级配置里选择目标摄像头设备和采集模式。
4. 若需要重试注册、停心跳、抓 SIP 报文，可在详情页继续使用手动按钮。
5. 点击开始推流后，收到 INVITE 时会自动启动本地摄像头码流发送。
6. 目录查询、设备信息查询能收到响应。
7. 错误密码时能看到明确认证失败原因。
8. 编辑 `国标设备 ID` 后，下一次连接会按新身份重新绑定平台视频资产。
9. 手动断开或注销后，不应再在 32 秒后追加 `REGISTER(unregister) transaction timeout` 噪声日志。

### 3.2 RTSP / RTMP

1. 选择媒体源为 `本地摄像头` 时，系统按当前环境的 `ZLM 推流主机 / RTSP 端口 / RTMP 端口` 自动生成无凭据 `sourceUrl` 并回填平台资产。
2. 本地摄像头模式固定要求启用统一认证；模拟器会在运行时推流地址里追加 `authUser/authPass`，但界面与平台资产仍只展示无凭据地址。
3. 同一场景下，设备表单会自动枚举本机 IPv4 并回填 `设备 IP`；平台资产列表中的 IP 列应显示模拟器主机地址，而不是 ZLM 基础设施地址。
4. 点击 `开始推流` 时，模拟器会先等待本地 ffmpeg 实际开始向 ZLM 发送码流，再调用平台 `startStream`，避免源地址尚未就绪就触发平台拉流。
5. 如有多路本机摄像头，先在设备高级配置里选择目标摄像头设备和采集模式。
6. 选择媒体源为 `外部源地址` 时，需要填写无凭据 `sourceUrl`；若地址里内嵌用户名密码，模拟器会直接拒绝。
7. 首次连接创建平台设备资产。
8. 再次连接优先复用同一 `platformDeviceId`。
9. 若编辑后切换了视频模式、媒体源类型或外部源地址，旧 `platformDeviceId` 会被清空，重新连接时按新身份重新查重。
10. 同一台 `RTSP / RTMP` 设备重复点击开始推流时，平台会优先复用现有 `live/{streamId}` 会话，不应再直接报 `This stream already exists`。

## 4. 常见问题

### 4.1 Video 设备提示未登录当前环境

先检查左上角环境是否已登录，再检查网关是否能访问 `DEVICE` 和 `MEDIA` 两类路由。

### 4.2 连接成功但重复创建设备

排查：

1. 检查模拟器版本是否已使用 `platformDeviceId`
2. 检查是否先走了 `deviceVideoList`
3. 检查平台唯一键约束是否最新

### 4.3 设备资产同步成功但媒体控制失败

排查：

1. 检查 `MEDIA` 路由是否可达
2. 检查 `platformDeviceId` 是否已回填
3. 检查 `firefly-media` 是否已部署到控制型视频接口版本

### 4.4 GB28181 已注册但播放超时/网络错误

排查：

1. 检查日志中是否出现 `本地码流异常：本地码流启动失败`
2. 若日志出现多条 `本地码流提示：检测到摄像头参数不兼容，改用 ... 重试`，说明系统已根据本次失败日志里的实际被拒绝模式，按摄像头 `Supported modes` 顺序尝试兼容参数
3. 若日志出现 `本地码流提示：Selected pixel format ...`，以及后续 `Supported pixel formats` 列表明细，说明系统已识别到输入像素格式不兼容，会继续按 `Supported pixel formats` 自动重试
4. 若日志继续出现 `本地码流提示：检测到摄像头参数不兼容，已切换设备默认采集模式重试`，说明已进入第二阶段降级，可继续观察是否恢复推流
5. 检查模拟器摄像头权限是否已授权
6. 检查 SIP INVITE 日志是否携带正确的 RTP 目标 `ip:port`
7. 检查媒体服务是否可达并已开放接收端口

### 4.5 RTSP / RTMP 本地摄像头推流失败

排查：

1. 检查当前环境里的 `ZLM 推流主机 / RTSP 端口 / RTMP 端口` 是否指向真实可达的 ZLM 地址，而不是遗留的 `localhost:554`。
2. 检查设备是否已启用认证，且 `authUsername/authPassword` 已填写完整；本地摄像头模式缺少认证信息会在开始推流前直接失败。
3. 检查平台资产里的 `sourceUrl` 是否为自动生成的无凭据地址。
4. 检查平台设备资产的 `ip` 是否已显示为模拟器主机地址，而不是 ZLM 宿主机地址。
5. 若日志改为 `本地码流已启动但推流地址尚未就绪`，优先检查 ZLM 地址可达性和目标端口监听状态。
6. 若媒体服务日志出现 `on_publish/on_play` 认证失败，优先核对运行时 `authUser/authPass` 与设备统一认证字段是否一致。
7. 检查 `开始推流` 时是否出现 `本地摄像头推流已启动` 日志。
8. 若日志出现 `Selected pixel format ... is not supported`，优先观察后续是否已有 `改用 ... 重试` 或 `已切换设备默认采集模式重试` 提示。
9. 检查媒体服务是否可达自动生成地址中的主机和端口。

### 4.6 GB28181 断开后 32 秒又提示 REGISTER(unregister) transaction timeout

排查：

1. 当前版本的模拟器在断开/注销时已改成 best-effort 注销，不应再补打一条延迟 32 秒的超时日志。
2. 若仍能看到该日志，说明本机运行的还是旧模拟器包，需要重新构建或替换安装包。

### 4.7 模拟器启动时报 ffmpeg 运行时缺失

排查：

1. 在 `firefly-simulator` 目录执行 `npm i`，确认 `@ffmpeg-installer/ffmpeg` 及当前平台子包已安装。
2. 执行 `npm run build:vite`，确认 Electron 主进程打包成功。
3. 如仍报错，设置环境变量 `FFMPEG_PATH` 指向本机可执行 ffmpeg 文件后再启动模拟器。

### 4.8 编辑后为什么要重新连接

排查：

1. 模拟器连接建立后，认证、Socket、SIP、平台视频资产绑定都已落到运行态。
2. 编辑保存只更新持久化配置，不会对现有连接做“热切换”。
3. 当前实现会在保存已连接设备前先执行断开，保存完成后需手工重新连接，确保新配置和运行态一致。

### 4.9 详情页为什么还看到 15fps

排查：

1. 打开 Video 详情页参数弹窗，确认 `采集模式` 已改成目标模式。
2. `GB28181` 场景下，采集参数会同步到 SIP 本地媒体配置，用于后续 INVITE 推流。
3. `RTSP / RTMP` 场景下，如当前已在推流，修改采集参数后需要停止再重新开始推流。
