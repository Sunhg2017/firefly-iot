# 告警管理运维说明

## 1. 模块说明

告警管理由 `firefly-rule` 和 `firefly-web` 协同提供，当前已拆分为两个独立入口：

- 告警规则维护：规则配置、启停和删除
- 告警处理：告警记录确认、处理和关闭

本次变更不涉及数据库结构变更，也不新增配置中心参数。

## 2. 部署影响

### 2.1 后端

- 模块：`firefly-rule`
- 影响内容：告警记录列表与详情接口权限改为 OR 授权

### 2.2 前端

- 模块：`firefly-web`
- 影响内容：
  - 菜单拆分为两个入口
  - 路由权限支持 `string[]`
  - 旧 `/alarm` 路径兼容跳转

## 3. 发布步骤

### 3.1 后端发布

```bash
mvn -pl firefly-rule -DskipTests compile
```

确认通过后，按常规流程发布 `firefly-rule` 服务。

### 3.2 前端发布

```bash
cd firefly-web
npm run build
```

确认构建通过后，按常规流程发布前端静态资源。

## 4. 权限与菜单检查

发布后需重点检查角色菜单是否符合预期。

### 4.1 规则维护角色

建议至少具备：

- `alarm:read`
- `alarm:create`
- `alarm:update`
- `alarm:delete`

### 4.2 告警处理角色

建议至少具备：

- `alarm:confirm`
- `alarm:process`

如处理角色需要查看更多上下文，也可补充：

- `alarm:read`

但本次改造后，即使不补 `alarm:read`，处理角色也能访问告警记录列表和详情。

## 5. 验收清单

发布完成后按以下清单验收：

1. 登录规则维护角色，确认只看到“告警规则维护”或同时看到两个菜单。
2. 登录处理角色，确认至少能看到“告警处理”菜单。
3. 处理角色访问 `/alarm-records` 不返回 403。
4. 处理角色访问 `/alarm-rules` 返回 403 或菜单不可见。
5. 访问旧 `/alarm` 路径时自动跳到 `/alarm-records`。

## 6. 日志与排障

### 6.1 前端菜单不显示

排查方向：

1. 检查当前登录用户的权限集合是否包含 `alarm:read`、`alarm:confirm`、`alarm:process` 之一。
2. 检查租户空间菜单配置是否隐藏了 `/alarm-rules` 或 `/alarm-records`。
3. 清理浏览器缓存后重新登录，确认用户权限和菜单树已刷新。

### 6.2 页面中文显示异常

排查方向：

1. 确认前端已发布到包含告警文案集中治理的最新版本。
2. 强制刷新浏览器缓存，避免继续加载旧的告警页面资源包。
3. 如仅个别浏览器异常，优先检查本地代理、响应头或静态资源网关是否错误改写编码。

### 6.3 打开告警处理页返回 403

排查方向：

1. 检查后端是否已发布到包含本次 OR 授权修改的版本。
2. 检查用户是否实际拥有 `alarm:confirm` 或 `alarm:process`。
3. 查看网关和 `firefly-rule` 服务日志中是否存在 `PERMISSION_DENIED`。

### 6.4 旧 `/alarm` 路径失效

排查方向：

1. 确认前端静态资源是否已更新到包含兼容跳转的版本。
2. 确认浏览器未命中旧缓存包。

## 7. 回滚方案

若需回滚，请同时回滚前后端版本：

1. 回滚 `firefly-web` 到拆菜单前版本。
2. 回滚 `firefly-rule` 到仅 `alarm:read` 控制记录读取的版本。

不要只回滚单侧：

- 只回滚前端会导致新权限模型无法被界面正确呈现。
- 只回滚后端会导致处理角色能看到菜单但进入记录页后 403。
