# 告警管理运维说明

## 1. 模块说明

告警管理已拆分为两个菜单：

- 告警规则维护
- 告警处理

本次发布主要影响“告警规则维护”能力，核心变化如下：

1. 告警规则统一使用结构化 JSON。
2. 一条规则可维护多个级别块。
3. 每个级别块支持 `全部满足`、`任意满足`、`至少 N 条满足` 三种触发语义。
4. 规则主级别由系统自动推导，不再由用户单独维护。

## 2. 发布影响

### 2.1 前端

需要发布 `firefly-web`，新增/调整内容包括：

- 告警规则维护页改为级别块式编辑器
- 页面中文文案与预览逻辑调整
- 按项目/产品/设备/物模型属性联动选择
- 规则列表摘要改为规则组自然语言描述

### 2.2 后端

需要发布 `firefly-rule`，新增/调整内容包括：

- 只接受结构化 `conditionExpr`
- 保存时校验 `groups[]` 结构
- 校验 `triggerMode` 和 `matchCount`
- 根据所有级别块自动推导规则主级别

### 2.3 数据库

本次无数据库迁移，无新增表、无新增字段。

## 3. 发布前检查

### 3.1 编译检查

后端：

```bash
mvn -pl firefly-rule -DskipTests compile
```

前端：

```bash
cd firefly-web
npm run build
```

### 3.2 协议检查

发布前必须确认下游是否已接受新的 `conditionExpr` 结构：

```json
{
  "mode": "STRUCTURED",
  "version": 3,
  "groups": [
    {
      "level": "WARNING",
      "triggerMode": "ALL",
      "conditions": []
    }
  ]
}
```

如果仍有任何下游任务、脚本、执行器按旧文本规则或旧 `conditions[]` 模型解析，需要先完成适配再发布。

## 4. 发布步骤

1. 发布 `firefly-rule`
2. 验证告警规则增删改查接口
3. 发布 `firefly-web`
4. 使用规则维护账号登录，验证“告警规则维护”页面
5. 使用处理账号登录，验证“告警处理”页面

## 5. 发布后验收

### 5.1 页面验收

1. “告警规则维护”和“告警处理”菜单正常分离。
2. 页面中文显示正常，无乱码。
3. 新建规则时可新增多个级别块。
4. 每个级别块都能先选择级别，再选择触发语义。
5. `AT_LEAST` 模式下可填写满足条数。
6. 每个级别块下可维护多条条件。
7. 支持阈值、同比/环比、连续触发、累计聚合、自定义表达式。
8. 规则列表能展示范围、级别标签、触发类型和摘要。

### 5.2 接口验收

1. 新建规则成功。
2. 编辑规则成功。
3. 同一规则多个级别块保存成功。
4. 非结构化 JSON 被拒绝。
5. `AT_LEAST` 且 `matchCount` 大于条件数时被拒绝。
6. 条件缺项时接口返回参数错误。

## 6. 监控与日志

### 6.1 重点日志

`firefly-rule` 发布后重点关注：

- `告警触发条件不能为空`
- `告警规则必须使用结构化条件`
- `告警规则必须使用合法的结构化 JSON`
- `告警规则至少需要维护一个级别块`
- `第 X 个级别块缺少触发语义`
- `满足条数不能大于条件总数`
- 其他 `PARAM_ERROR`

### 6.2 重点现象

1. 页面保存时报参数错误。
2. 规则打开后无法正确回填级别块。
3. 产品已选择但物模型属性为空。
4. 规则主级别与级别块配置不一致。

## 7. 常见故障排查

### 7.1 页面没有项目、产品、设备可选

排查步骤：

1. 检查当前租户下是否已有项目、产品、设备数据。
2. 检查项目、产品、设备列表接口是否正常。
3. 检查当前账号是否具备读取权限。

### 7.2 选择产品后没有指标

排查步骤：

1. 检查产品是否已维护物模型。
2. 检查产品物模型接口是否返回合法 JSON。
3. 检查 `properties[].identifier` 是否存在。

### 7.3 保存规则失败

排查步骤：

1. 检查是否至少存在一个级别块。
2. 检查每个级别块是否都选择了级别和触发语义。
3. 检查 `AT_LEAST` 的满足条数是否小于等于条件数。
4. 检查每个级别块下是否至少有一条条件。
5. 检查条件类型对应字段是否填写完整。
6. 检查浏览器请求体中的 `conditionExpr` 是否为合法 JSON。

### 7.4 规则主级别看起来不对

这是系统设计行为。

说明：

- 主级别不是手填字段
- 主级别取所有级别块中的最高严重等级
- 用于列表筛选和统计

排查步骤：

1. 检查各级别块的 `level` 配置。
2. 检查前端发送的 `groups[]` 是否完整。
3. 检查后端返回的 `level` 是否已更新。

### 7.5 下游执行链路不识别新规则

原因：

- 当前规则协议已经统一到 `groups[]`
- 不再兼容旧文本规则
- 下游如果仍按旧结构消费会失败

处理建议：

1. 检查下游是否直接解析 `conditionExpr`。
2. 将解析逻辑升级为 `version = 3` 的结构化规则。
3. 按级别块和触发语义实现命中计算。

## 8. 回滚方案

### 8.1 代码回滚

若需要回滚，请同时回滚：

1. `firefly-web`
2. `firefly-rule`

### 8.2 风险说明

虽然本次没有数据库迁移，但如果线上已经创建了新结构规则，回滚后旧版本可能无法理解 `groups[]` 协议。

回滚前建议：

1. 导出或记录新建规则。
2. 确认回滚版本是否能解析结构化规则。
3. 若不能解析，优先回滚整个发布而不是仅回滚单侧服务。
