# 前端根路由首页跳转修复运维说明
## 1. 适用范围

本说明适用于 `firefly-web` 首页路由修复后的发布、验证与问题排查。

## 2. 发布影响

本次仅影响前端路由跳转行为，不涉及：

- 数据库结构变更
- Flyway SQL 执行
- 菜单台账补录
- 后端接口发布顺序调整

受影响模块：

- `firefly-web`

## 3. 发布步骤

1. 在前端工程执行构建校验：

```bash
cd firefly-web
npm run build
```

2. 发布最新前端静态资源。

## 4. 发布后验证

### 4.1 根路由验证

- 使用一个没有 `/dashboard` 权限、但拥有其他菜单权限的账号登录。
- 直接访问系统根路径 `/`。
- 预期结果：跳转到该账号首个可访问页面，而不是 `/403`。

### 4.2 一致性验证

- 登录成功后的自动跳转应与直接访问 `/` 的结果一致。
- 在 `/403` 页面点击“返回首页”，应回到同一个首页。
- 输入一个不存在的站内路径后，系统应回到当前用户首页，而不是固定跳到 `/dashboard`。
- 直接访问一个已注册但当前账号无权访问的业务路径时，系统仍应进入 `/403`。

## 5. 排查方法

如果用户访问 `/` 仍进入 `/403`，按以下顺序排查：

1. 检查 `/users/me` 返回的 `authorizedMenuPaths` 是否为空。
2. 检查 `/users/me/permissions` 是否包含对应业务页面所需权限。
3. 检查该用户是否真的存在至少一个可访问菜单。
4. 检查前端静态资源是否包含本次 `BasicLayout` 布局放行修复，而不是仍在使用旧包。

如果三个条件都不满足，进入 `/403` 属于正常授权结果，而不是本次修复失效。

## 6. 回滚说明

本次不建议回滚到固定跳转 `/dashboard` 的旧逻辑，因为旧逻辑会重新引入“用户明明有页面权限，但访问根路径直接 403”的问题。

如必须回滚代码版本，只回滚前端资源即可；数据库和后台服务无需回滚。
