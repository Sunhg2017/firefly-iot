# 列表查询交互统一设计

## 背景

当前多个列表页存在两类问题：

- 查询按钮样式、位置不统一，有的和输入框绑定在一起，有的直接挤在筛选项中间。
- 一部分列表把筛选项状态直接挂在请求依赖上，导致用户改一次条件就自动查询一次。

这会带来两个直接问题：

1. 用户在连续调整多个筛选项时会触发多次无意义请求。
2. 不同页面的查询区行为不一致，难以形成稳定使用预期。

## 目标

- 列表页首次进入时自动加载一次。
- 首次加载之后，筛选条件只在点击 `查询` 时生效。
- 查询区统一收口为“筛选项在前、操作按钮在后”的布局。
- 已生效筛选在翻页时继续沿用，不因草稿改动而自动重查。

## 适用范围

本次收口覆盖以下核心列表页：

- `设备管理`
- `视频监控`
- `产品接入`
- `设备联动规则`
- `设备拓扑`
- `审计日志`
- `异步任务中心`
- `定时任务管理`
- `自定义协议解析`
- `OTA 升级`
- `设备数据 -> 设备事件`

## 交互模型

### 1. 两层筛选状态

每个列表页统一拆成两层状态：

- `draftFilters`
  - 页面控件当前值，只负责承接用户正在编辑的筛选条件。
- `filters`
  - 当前真正参与后端查询的已生效条件。

### 2. 查询触发规则

- 页面首次渲染时，使用默认 `filters` 自动查询一次。
- 用户修改输入框、下拉框、级联筛选时，只更新 `draftFilters`。
- 用户点击 `查询` 时，才把 `draftFilters` 提交到 `filters`。
- 用户点击 `重置` 时，同时清空 `draftFilters` 与 `filters`，并重新查询第一页。
- 用户切换分页时，继续沿用当前 `filters`。

### 3. 查询区布局

统一使用：

- `ff-query-card`
- `ff-query-bar`
- `ff-query-actions`

布局规则：

- 输入项在左侧连续排列。
- `查询 / 重置 / 新建` 等操作按钮固定放在尾部动作区。
- 查询按钮不再嵌入 `Input.Search` 右侧附属按钮中。

## 实现要点

### 前端

- 公共样式落在 `firefly-web/src/styles/global.css`
- 列表页内部统一按“草稿筛选 + 已生效筛选”模式改造
- `Query` 按钮负责触发 `setFilters(...)`，而不是在 `onChange` 里直接请求

### 后续新增页面约束

- 新增列表页默认复用这套交互模型
- 禁止再写“筛选项一改就请求”的实现
- 禁止把 `查询` 按钮夹在筛选项中间，或重新退回到 `Input.Search enterButton`

## 风险与取舍

- 旧页面如果仍保留历史写法，交互会再次分裂，因此后续新增或重构列表页时必须继续按本设计执行。
- 已生效筛选与草稿筛选分离后，表格刷新、分页、导出等动作必须明确依赖 `filters`，不能错误读取草稿值。
