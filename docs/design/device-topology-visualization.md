# 设备拓扑可视化设计说明
## 1. 背景

现有控制台虽然能管理设备、分组、标签和影子，但缺少一套专门展示父子设备关系的视图。对于网关与子设备场景，运维只能回到设备列表靠筛选和字段比对判断挂载关系，无法快速识别：

- 某台子设备挂在哪个网关下
- 某个网关当前挂了多少下级设备
- 设备是否已经断链，引用了不存在或不可见的上级设备
- 当前筛选命中的设备位于哪条链路里

因此需要补一个独立的“设备拓扑”能力，把设备关系从普通列表里分离出来，收口为专门的拓扑查询接口和可视化页面。

## 2. 目标

- 在租户空间提供独立的“设备拓扑”页面，展示父子设备链路、独立设备和断链设备。
- 查询时保留上下游链路上下文，避免筛选后把真实关系裁断成伪孤点。
- 复用现有 `devices.gateway_id` 与 `products.node_type`，不新增关系表，不引入双轨模型。
- 菜单、路由、权限口径统一收口到当前实现，复用已有 `device:read`。

## 3. 范围

本次改动包含：

- `firefly-device` 新增设备拓扑查询接口和 DTO
- `firefly-web` 新增设备拓扑独立页面、路由、菜单配置和设备管理入口
- `firefly-system` 新增 Flyway SQL，维护 `workspace_menu_catalog` 与 `workspace_menu_permission_catalog`

本次不包含：

- 新增设备关系表或兼容旧关系模型
- 新增独立权限编码
- 修改设备注册、影子、消息下发等其他设备子模块

## 4. 数据模型

### 4.1 关系来源

- 父子关系：`devices.gateway_id`
- 节点类型：`products.node_type`
- 设备状态：`devices.status`
- 在线状态：`devices.online_status`

### 4.2 拓扑节点输出

接口输出 `DeviceTopologyNodeVO`，包含：

- 设备名称、别名
- 产品名称、`productKey`
- 节点类型 `GATEWAY / DEVICE`
- 上级网关名称
- 直属子设备数、子树节点数
- 在线状态、设备状态、固件版本、IP
- `matched`：是否命中当前筛选
- `orphan`：是否为断链节点

### 4.3 拓扑分类

- `rootNodes`：拓扑根节点。规则为 `gatewayId = null` 且存在下级节点。
- `standaloneDevices`：独立设备。规则为 `gatewayId = null` 且没有下级节点。
- `orphanDevices`：断链设备。规则为 `gatewayId != null` 但当前可见范围内找不到上级节点。

## 5. 查询策略

### 5.1 入口

- 后端接口：`POST /api/v1/devices/topology`
- 权限：`device:read`

### 5.2 筛选字段

- `keyword`
- `productId`
- `groupId`
- `projectId`
- `status`
- `onlineStatus`

### 5.3 上下游保留规则

拓扑查询与普通设备列表不同，不直接把筛选结果当最终展示结果，而是分两步处理：

1. 先按筛选条件找出命中节点。
2. 再补齐这些节点的祖先链路和后代链路。

这样可以保证：

- 命中子设备时，仍然能看到它挂载在哪个网关下。
- 命中网关时，仍然能看到整棵子树。
- 断链节点会被正确归入断链区域，而不是混入普通根节点。

## 6. 前端方案

### 6.1 页面布局

- 页头：展示刷新、展开全部、收起到首层操作。
- 概览卡片：命中设备、当前展示、根节点、网关设备、独立设备、断链设备。
- 筛选区：产品、分组、设备状态、在线状态、关键字。
- 主视图区：
  - 左侧 `Tabs`
    - 拓扑链路
    - 独立设备
    - 断链设备
  - 右侧节点详情

### 6.2 拓扑链路

- 使用 Ant Design `Tree` 展示层级关系。
- 节点卡片中同时展示：
  - 别名 / 设备名称
  - 节点类型
  - 在线状态
  - 设备状态
  - 产品 `productKey`
  - 下级数量
- 命中筛选的节点用蓝色边框突出。

### 6.3 详情区

详情区仅展示当前操作所需信息：

- 设备名称
- 产品名称与 `productKey`
- 当前链路位置
- 在线状态与设备状态
- 直属下级、子树规模
- 固件版本、IP、最后在线时间
- 当前路径

## 7. 菜单与权限

新增租户空间菜单：

- 父菜单：`tenant-device-assets`
- 菜单键：`device-topology`
- 路由：`/device-topology`
- 权限：`device:read`

通过 Flyway `V32__add_device_topology_menu.sql` 维护：

- `workspace_menu_catalog`
- `workspace_menu_permission_catalog`

本次不新增 `permission_resources`，因为复用了现有 `device:read`。

## 8. 风险与约束

- 拓扑查询会递归补齐祖先和后代，为避免一次性拉取过大范围，服务端增加 3000 节点保护阈值；超过阈值时要求先缩小筛选范围。
- 如果历史数据里存在错误自环或环形关系，服务端会跳过递归死循环并记录日志。
- 如果设备关系跨项目且当前按项目筛选，超出项目范围的上级节点不会被补齐，子设备会按断链显示。

## 9. 验证点

- `firefly-device` 的 `DeviceServiceTest` 覆盖：
  - 命中子设备时保留祖先链路
  - 命中网关时展开整棵子树
- `firefly-web` 执行 `npm run build` 成功
- 设备管理页可跳转到设备拓扑页
- 菜单迁移执行后，租户空间出现“设备拓扑”菜单
