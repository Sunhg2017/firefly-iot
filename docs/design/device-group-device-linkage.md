# 设备分组与设备主流程联动设计

## 背景

当前仓库中已经存在设备分组能力，但原实现与设备主流程存在明显断层：

- 设备创建、编辑、批量导入时无法直接维护分组
- 设备删除后不会自动清理分组成员关系
- 设备列表和设备详情无法直接看到结构化分组信息
- 设备分组页面保留了未启用的动态分组入口，容易造成理解和运维歧义

本次改造以“功能未正式启用、不需要兼容旧配置”为前提，直接收口为静态分组方案，并把设备分组接入现有设备管理主链路。

## 目标

- 设备创建、编辑、批量导入时可直接选择分组
- 设备列表、设备详情返回结构化 `groupList`
- 设备列表支持按分组筛选
- 删除设备时自动清理分组成员关系并回刷分组设备数
- 设备分组页仅保留静态分组，去掉未启用的动态分组维护入口

## 范围

### 后端

- `firefly-device`
  - 设备相关 DTO 增加 `groupIds`
  - 设备查询 DTO 增加 `groupId`
  - 设备详情/列表返回 `groupList`
  - 设备服务接入分组同步、删除清理、按分组筛选
  - 设备分组服务收口为静态分组，并补齐成员同步、递归删除等能力

### 前端

- `firefly-web`
  - 设备管理页增加分组选择、分组展示、按分组筛选
  - 设备分组页只保留静态分组交互
  - 批量导入设备时支持统一分组

## 核心设计

## 1. 分组模型收口为静态分组

- `DeviceGroupService` 统一限制 `type=STATIC`
- 创建、编辑、树查询、列表查询都仅面向静态分组
- 动态规则字段不再作为有效配置入口，统一置空

这样做的原因是当前动态分组没有完整执行链路，继续暴露只会让页面、接口和运维行为不一致。

## 2. 设备与分组关系统一由成员表维护

- 真实关系表：`device_group_members`
- 设备列表/详情中的 `groupList` 全部由成员表反查得到
- 设备删除时统一通过 `DeviceGroupService.removeDeviceMemberships` 清理

这保证了设备主流程和分组页面读到的是同一份关系数据，避免出现页面展示与实际分组成员不一致。

## 3. 设备主流程统一维护 `groupIds`

下列 DTO 统一接入 `groupIds`：

- `DeviceCreateDTO`
- `DeviceUpdateDTO`
- `DeviceBatchCreateDTO`
- `DeviceImportDTO`

设备服务在以下场景调用 `DeviceGroupService.syncDeviceGroups`：

- 创建设备
- 编辑设备
- 批量创建设备
- 批量导入设备

## 4. 设备列表支持按分组筛选

- `DeviceQueryDTO` 新增 `groupId`
- `DeviceService.buildListWrapper` 在查询条件中接入分组过滤
- 过滤实现方式：
  - 先从 `device_group_members` 查询目标分组下的设备 ID
  - 再将设备 ID 限定到设备分页查询条件中

该实现不引入新的 Mapper SQL，保持现有分页查询链路稳定。

## 5. 分组树删除采用递归清理

删除分组时：

1. 递归收集当前分组及所有子分组
2. 批量删除对应成员关系
3. 批量删除分组节点

这样可以保证分组树结构删除后不会残留孤儿成员数据。

## 接口与数据变化

### 设备接口

- 创建设备：支持 `groupIds`
- 编辑设备：支持 `groupIds`
- 批量导入设备：支持 `groupIds`
- 设备列表查询：支持 `groupId`
- 设备列表/详情：返回 `groupList`

### 设备分组接口

- 查询树、列表、详情：仅返回静态分组
- 新建/编辑：仅允许静态分组
- 删除：支持递归删除当前分组及其子分组

## 测试策略

新增 `DeviceGroupServiceTest`，覆盖两条关键链路：

- `syncDeviceGroups` 会替换成员关系并刷新设备数
- `deleteGroup` 会递归删除子分组与成员关系

## 风险与控制

- 风险：旧的动态分组入口仍被前端或接口误调用
  - 控制：服务层统一限制仅支持 `STATIC`
- 风险：设备删除后分组成员残留
  - 控制：删除设备时强制执行成员清理
- 风险：列表分页和分组筛选组合后结果为空
  - 控制：当分组下无设备时，查询条件直接返回空结果，不再走无意义分页
