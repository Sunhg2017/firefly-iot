# 产品设备接入体验优化设计

## 背景

产品接入页原来的设备接入能力分散在“查看密钥”“认证说明”“动态注册”等多个按钮里，用户需要自己判断先看哪个入口、再去哪一个页面继续操作。与此同时，动态注册结果直接展示了 `deviceId`、`productId`，不符合仓库对“优先使用业务唯一键、避免暴露数据库主键”的约束。摄像头产品还存在两处割裂：产品页仍显示普通 IoT 认证方式，而跳转到视频监控页时也没有带上产品上下文。

## 目标

- 将产品侧设备接入入口收敛为单一“设备接入”入口。
- 在同一抽屉中完成接入路径说明、协议参数查看、动态注册调试和后续动作引导。
- 对齐 HTTP、MQTT、CoAP 等协议的实际接入链路，避免页面说明与后端实现脱节。
- 清理动态注册对外返回和前端展示中的主键暴露。
- 收口摄像头产品认证口径，明确摄像头产品统一走“视频协议接入”。
- 打通产品页到视频监控页的联动，跳转后自动带入产品和协议上下文。

## 范围

本次改造覆盖以下范围：

- `firefly-web` 产品接入页中的设备接入入口和接入抽屉。
- `firefly-web` 视频监控页的“添加视频设备”抽屉联动。
- `firefly-connector` 动态注册 HTTP 接口的对外返回口径。
- `firefly-device` 摄像头产品认证口径收口。
- `firefly-simulator` 对动态注册结果的消费口径。

不在本次范围内：

- 内部 RPC `DeviceRegisterDTO` 结构。
- 设备管理页的创建、导入、三元组导出流程。
- `video_devices` 表与产品之间的持久化关联字段设计。

## 方案概述

### 1. 入口收敛

产品列表不再分散提供“查看密钥 / 认证说明 / 动态注册”多个入口，统一为“设备接入”按钮。这样用户只需进入一个入口，即可根据当前产品认证方式得到后续动作。

### 2. 抽屉重构

设备接入抽屉重构为三个任务化标签页：

- 接入指引：展示当前产品的接入路径、认证约束、ProductSecret 或手动创建设备说明。
- 协议参数：按产品协议给出 MQTT / HTTP / CoAP / 其他协议的实际接入参数、Topic、接口和注意事项。
- 动态注册：仅对一型一密产品开放，开发中和已发布状态都支持直接调试动态注册并展示生成的业务口径结果。

### 3. 协议说明和实际实现对齐

接入说明直接对齐现有 connector 能力：

- MQTT：`ClientId = productKey.deviceName`，`Username = deviceName&productKey`，密码最终统一为 `DeviceSecret`。
- HTTP：先 `/api/v1/protocol/http/auth` 换取 `X-Device-Token`，再调用属性、事件、心跳接口。
- CoAP：先 `/api/v1/protocol/coap/auth`，再通过 token 调属性、事件、OTA 进度接口。
- 其他协议：提示优先结合协议解析和设备模拟器联调。

### 4. 主键暴露治理

动态注册接口对外响应从：

- `deviceId`
- `productId`
- `deviceName`
- `deviceSecret`

调整为：

- `productKey`
- `deviceName`
- `deviceSecret`

前端抽屉只展示业务唯一键和接入凭证，不再向用户暴露数据库主键。

### 5. 摄像头产品认证收口

摄像头产品不再复用普通 IoT 的 `一机一密 / 一型一密` 表达。页面统一展示为：

- 认证方式：`视频协议接入`
- 接入路径：跳转 `视频监控`
- 禁止项：不再展示 `ProductSecret`，不再开放动态注册

后端仍使用 `DEVICE_SECRET` 作为内部存储占位，以避免扩大枚举和数据库兼容面，但对用户界面统一呈现为 `视频协议接入`。同时清理摄像头产品的历史 `product_secret`，避免继续把摄像头误识别为一型一密产品。

### 6. 产品跳转视频监控联动

产品页点击摄像头产品的“视频接入”后，前端会把以下上下文带到 `/video`：

- `productId`
- `productKey`
- `productName`
- `protocol`
- `autoCreate=1`

视频监控页收到上下文后执行两步：

1. 自动打开“添加视频设备”抽屉
2. 将接入方式锁定为产品协议（GB28181 / RTSP / RTMP）

当前联动范围仅覆盖前端跳转和表单预填，不在本次范围内引入 `video_devices` 与产品的持久化外键字段。

## 页面与交互流程

### 一机一密

1. 用户在产品列表点击“设备接入”。
2. 接入指引页明确提示：需先前往设备管理页创建或导入设备。
3. 协议参数页提供实际协议连接方式。
4. 动态注册页显示“不支持动态注册”，并给出“去设备管理创建设备”动作。

### 一型一密

1. 用户在产品列表点击“设备接入”。
2. 接入指引页展示 ProductSecret、接入步骤和后续动作。
3. 协议参数页说明动态注册后如何按具体协议接入。
4. 动态注册页直接调试注册，并展示 `productKey / deviceName / deviceSecret`。

### 摄像头产品

1. 用户在产品列表点击“视频接入”。
2. 抽屉明确提示当前产品不走 `ProductSecret` 和动态注册。
3. 点击“去视频监控”后，页面自动跳转到 `/video` 并打开“添加视频设备”抽屉。
4. 视频监控页自动锁定接入方式为产品协议，用户只需补充设备名称、地址、厂商型号等字段。

## 接口口径

### 动态注册对外接口

- 路径：`POST /api/v1/protocol/device/register`
- 请求：
  - `productKey`
  - `productSecret`
  - `deviceName`
  - `nickname`
  - `description`
  - `tags`
- 响应：
  - `productKey`
  - `deviceName`
  - `deviceSecret`

### 产品密钥查看

- 路径：`GET /api/v1/products/{id}/secret`
- 仅一型一密产品可查看。

## 设计取舍

- 保留抽屉形态而非跳转独立页面，是为了保证产品上下文不丢失，用户可以在产品列表就地完成接入准备。
- 协议说明优先给“可直接复制”的参数、Topic、接口路径，而不是抽象概念，降低联调成本。
- 动态注册结果不再返回对外无意义的主键，但内部服务之间仍可继续使用 `DeviceRegisterDTO` 中的 ID 字段。
- 摄像头产品暂不新增数据库级产品关联字段，先把跳转联动和协议锁定做好，避免在未确定最终关系模型前继续扩散半成品设计。

## 风险与注意事项

- 一型一密产品在开发中阶段也允许动态注册，便于设备联调和接入调试。
- ProductSecret 仍属于敏感信息，页面只允许查看和复制，不做持久缓存。
- 如果后续扩展更多协议，需要同步更新抽屉中的协议参数映射，保持页面说明与 connector 实现一致。
- 如果后续需要在视频设备列表中反查所属产品，应单独设计 `video_devices` 与产品的持久化关联字段和查询链路，而不是继续依赖 URL 参数。
