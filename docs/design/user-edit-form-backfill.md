# 用户编辑表单回填修复设计说明
> 模块: firefly-web / 用户管理
> 日期: 2026-03-17
> 状态: Done

## 1. 背景

用户管理页面改为抽屉表单后，编辑已有用户时出现“字段没有回填”的问题。管理员进入编辑态后，看不到用户名、姓名、手机号、邮箱和角色的当前值，容易误判为数据丢失，也无法安全修改角色。

## 2. 问题根因

- 用户编辑抽屉启用了 `destroyOnClose`，关闭后表单组件会被卸载。
- 旧实现里在抽屉真正挂载前就执行 `form.setFieldsValue(...)`，导致回填值在组件重新挂载时丢失。
- 列表接口返回的是摘要信息，编辑场景如果只依赖列表行数据，角色等字段不一定完整。
- 用户编辑抽屉是分步表单，但表单启用了 `preserve={false}`，导致未挂载步骤的字段值不会保留。角色字段位于第二步，编辑时即使详情接口返回了角色，也会在第一步阶段被丢弃，最终表现为角色没有回填。

## 3. 修复方案

- 新增 `toUserFormValues(...)`，统一把用户详情转换为表单值，避免编辑态和新建态初始化逻辑分散。
- 把表单初始化时机改为“抽屉打开后执行”，通过 `useEffect` 在抽屉挂载完成后再回填。
- 新建用户时统一在抽屉打开后重置表单，并初始化 `roles: []`。
- 编辑用户时先打开抽屉，再调用 `userApi.get(id)` 拉取用户详情，详情返回后再写入表单。
- 新增抽屉级加载态，避免详情尚未返回时页面闪动或用户误操作。
- 移除 `Form` 上的 `preserve={false}`，让分步字段在切步和延迟挂载时仍保留在表单状态里，保证角色回填、步骤切换和最终保存使用同一份数据。
- 如果详情接口失败，则回退使用列表数据回填，并给出明确提示，避免抽屉空白。

## 4. 影响范围

- `firefly-web/src/pages/user/UserList.tsx`

## 5. 设计取舍

- 没有保留“打开抽屉前预填值”的旧逻辑，直接收口为“挂载后初始化”的单一路径，避免再次出现时序不一致。
- 没有为列表摘要数据增加兼容分支，而是优先以详情接口作为编辑数据源，保证回填口径与保存口径一致。

## 6. 风险与约束

- 该方案依赖 `GET /api/v1/users/{id}` 正常返回用户详情与角色信息。
- 如果后续用户详情字段继续扩展，需要同步更新 `toUserFormValues(...)`，确保新增字段也能回填。
