# 设备中心菜单结构优化设计

## 背景

设备中心左侧菜单存在两个明显问题：

1. 产品页页面标题已经收口到“产品接入”，但租户菜单配置仍可能展示旧名称“产品管理”。
2. `SNMP / Modbus / WebSocket / TCP/UDP / LoRaWAN` 等协议入口在设备中心内平铺展示，菜单过长，认知负担较大。

同时，前端默认路由和租户自定义菜单使用的是两套不同的菜单拼装逻辑，导致改了默认路由后，租户已保存的菜单结构仍可能显示旧口径。

## 目标

- 统一设备中心菜单中“产品接入”的命名口径。
- 将协议类入口抽成“协议接入”分组，减少菜单平铺长度。
- 让默认菜单、租户菜单配置、租户空间菜单分配都基于同一套递归菜单树能力。
- 在不强制清空历史租户菜单配置的前提下，兼容已有菜单数据并修正展示。

## 方案概述

## 1. 路由树改为递归结构

`routes.tsx` 中的 `RouteGroup.children` 从仅支持叶子路由改为支持 `RouteNode[]`，允许设备中心下继续挂子分组。

本次新增的结构为：

- 设备中心
  - 产品接入
  - 协议解析
  - 设备管理
  - 设备分组
  - 设备标签
  - 地理围栏
  - 设备影子
  - 设备消息
  - 协议接入
    - SNMP 接入
    - Modbus 接入
    - WebSocket 接入
    - TCP/UDP 接入
    - LoRaWAN 接入

## 2. 布局层改为递归菜单渲染

`BasicLayout` 的默认菜单、租户菜单、面包屑和展开态计算统一改为递归遍历，不再假设“只有一级分组 + 一级叶子菜单”。

这样可以保证：

- 默认路由树可以正确渲染二级分组
- 面包屑可以显示 `设备中心 / 协议接入 / SNMP 接入`
- 菜单展开态能准确定位到多级父节点

## 3. 兼容旧租户菜单配置

考虑到现网租户可能已经保存了旧菜单配置，本次不要求用户手工重建菜单，而是在前端渲染阶段做规范化：

- `/product` 强制按当前规范显示为“产品接入”
- `device-mgmt` 下若发现历史协议菜单平铺存在，则自动归并到合成的“协议接入”分组中
- 若租户菜单中已经有 `协议接入` 分组，则直接沿用并规范其标签与图标

该策略仅影响前端展示，不直接修改租户数据库中的菜单记录。

## 4. 菜单管理与空间分配同步收口

为了避免后续再次生成旧结构：

- 菜单管理页“初始化默认菜单”改为递归生成默认菜单树
- 租户空间菜单分配的候选树改为递归结构
- 路由路径选项展示父级路径链路，例如 `设备中心 / 协议接入 / SNMP 接入`

## 设计取舍

- 没有直接批量迁移租户菜单配置表，而是优先使用前端规范化展示，降低发布风险。
- 协议入口保留在设备中心下，而不是拆到新的顶级分组，避免设备接入链路被切散。
- 递归路由树一旦建立，后续设备中心继续新增子分组时无需再改布局基础能力。

## 风险与注意事项

- 旧租户菜单配置只做展示层归并，菜单管理页中如果手工编辑原始配置，仍应以最新默认结构为准。
- 如果后续新增新的协议接入页面，需要同步更新协议路径集合，确保归组逻辑完整。
- 递归菜单树涉及菜单、面包屑、空间分配三处联动，任何新增菜单结构都应按递归方式接入，避免回退到平铺假设。
