# 设备中心菜单结构优化设计

## 1. 背景

设备中心原先同时存在两类问题：

- 菜单命名不统一，“产品管理”和“产品接入”并存。
- 协议接入入口直接平铺在设备中心下，导航过长。

随着工作空间授权模型收口，菜单不再由租户侧自行拼装，而是统一由路由目录和权限目录驱动，因此菜单设计也需要同步收口。

## 2. 目标

- 统一设备中心信息架构和命名。
- 将协议类页面归入“协议接入”分组。
- 让设备中心在平台授权、租户展示、面包屑渲染上使用同一套路由目录。
- 不再为旧菜单结构保留前端归并和展示兜底。

## 3. 菜单结构

设备中心采用递归路由树：

- 设备中心
  - 产品接入
  - 协议解析
  - 设备管理
  - 设备分组
  - 设备标签
  - 地理围栏
  - 设备影子
  - 设备消息
  - 协议接入
    - SNMP 接入
    - Modbus 接入
    - WebSocket 接入
    - TCP/UDP 接入
    - LoRaWAN 接入

## 4. 实现原则

- `routes.tsx` 维护菜单层级与工作空间归属。
- `BasicLayout` 按递归树渲染侧边栏和面包屑。
- `workspaceRoutes.ts` 根据工作空间和权限过滤可见节点。
- 租户空间只展示平台已授权且当前用户有权限的菜单。

## 5. 关键取舍

- 不再为旧菜单结构做前端归并或重命名兼容。
- 如果数据库里仍残留旧菜单授权数据，应由系统运维重新授权或直接清理。
- 协议接入继续保留在设备中心内部，避免把设备接入链路拆散。

## 6. 验证要点

- 设备中心下可看到“协议接入”分组而不是平铺入口。
- “产品接入”名称在菜单、标题、面包屑中保持一致。
- 工作空间切换后，设备中心菜单仍能按当前权限正确展开与定位。
