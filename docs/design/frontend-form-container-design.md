# 前端表单容器设计规范

## 1. 背景

仓库中的配置型页面越来越多，很多页面已经不是几个字段的短表单，而是包含动态条件、联动下拉、分组配置、预览摘要和通知编排的复杂交互。如果继续把这类内容堆在普通弹窗里，容易出现以下问题：

1. 可视空间不足，表单滚动过长，用户很难定位当前正在配置的模块。
2. 配置过程中需要频繁回看列表和上下文，居中弹窗会打断连续操作。
3. 联动下拉、动态列表和复杂条件混在一个长抽屉中时，认知负担仍然过高。

因此，本仓库将“复杂表单使用抽屉”继续收敛为更明确的强制规则，并新增“长抽屉表单必须分步骤”的交互约束。

## 2. 强制规则

### 2.1 何时必须使用抽屉

满足以下任一条件时，创建、编辑、配置、调试类表单必须使用 `Drawer`：

- 字段数量明显超出短表单范围。
- 页面包含多个业务模块或多个配置区块。
- 包含 `Form.List`、条件表达式、脚本、JSON、规则块、调试参数等复杂结构。
- 用户需要边看列表上下文边配置。
- 页面需要同时展示预览、摘要、校验提示、运行结果等伴随信息。

### 2.2 何时可以继续使用弹窗

以下场景可以继续使用 `Modal`：

- 仅包含少量基础字段的短表单。
- 删除确认、二次确认、风险提示。
- 只读详情、小范围状态切换、短流程处理。

### 2.3 长抽屉表单的新增规则

当抽屉类表单出现以下情况时，必须拆分成按模块组织的步骤式配置：

- 抽屉内部出现多个明显独立的业务模块。
- 下拉选择、联动选择或动态配置区过长，用户需要频繁滚动。
- 单个抽屉内容长度已经影响用户理解当前配置位置。

步骤式抽屉必须满足以下要求：

1. 步骤按业务模块拆分，而不是机械按字段数量切段。
2. 底部必须提供“上一步 / 下一步”，最后一步才允许正式提交。
3. 已经完成或已到达的步骤必须支持回退访问，且已填写数据不能丢失。
4. 切换步骤时优先校验当前模块，不应让后续模块的未填写字段提前阻塞当前步骤。
5. 最终提交时仍需对整张表单做完整校验。

## 3. 当前落地范围

本次已将告警规则配置抽屉改为步骤式抽屉，作为仓库内的首个标准实现。步骤拆分如下：

1. 基本信息
2. 适用范围
3. 告警等级块
4. 通知编排
5. 预览与状态

当前检查结果：

- 告警规则抽屉属于典型长表单，已按本规范完成修复。
- 其他抽屉页面仍按“复杂度是否足以拆步骤”继续执行，不要求把所有抽屉一刀切改成步骤流。

## 4. 交互设计要求

### 4.1 步骤划分原则

- 一步只承载一个清晰的业务目标。
- 同一步内的字段应高度相关，避免跨模块混放。
- 步骤名称要直接表达用户当前正在配置什么。

### 4.2 数据保留原则

- 步骤切换必须复用同一份表单状态。
- 回到已完成步骤时，应能看到上次配置结果并继续修改。
- 不允许因为切换步骤而丢失已填写内容。

### 4.3 校验原则

- “下一步”只校验当前步骤对应模块。
- 最终“提交”校验整表。
- 涉及跨字段关系时，应给出模块内清晰错误提示。

### 4.4 输入控件原则

- 稳定可枚举的字段继续优先使用下拉、单选、多选、联动选择。
- 长抽屉拆步骤后，不应退化成大量手工输入框。
- 用户可见视角继续优先使用业务唯一键，不直接暴露数据库主键。

## 5. 设计取舍

### 5.1 为什么不是所有抽屉都拆步骤

步骤流会增加导航成本。如果抽屉本身只有一个简单模块，强行拆步骤会让操作变慢、信息更碎。因此步骤式抽屉只适用于真正的长表单和多模块表单。

### 5.2 为什么不是继续拉长抽屉

单纯把抽屉加宽或继续堆叠内容，只能缓解局部空间问题，不能解决“当前配置到哪里”“哪些模块已经完成”“哪些字段属于同一业务语义”的问题。

## 6. 后续约束

1. 新增复杂抽屉页面时，评审必须显式说明是否需要步骤化。
2. 已有抽屉如果继续膨胀为多模块长表单，必须同步改造成步骤式抽屉。
3. 文档、实现和验收口径必须保持一致，不能只改页面不更新规范说明。
