# 前端复杂表单容器设计说明

## 1. 背景

随着设备、规则、协议、任务等配置页面持续增加，平台内大量创建、编辑、调试、注册类页面已经不再是“少量字段录入”，而是包含多分区、多步骤、联动选择、长文本配置和 JSON/脚本配置的复杂表单。继续使用弹窗会带来以下问题：

1. 可视区域过小，长表单容易挤压、遮挡和滚动混乱。
2. 用户在填写时无法同时保留列表上下文，不利于比对和回填。
3. 复杂配置需要频繁试填、预览、修改时，弹窗的关闭和重新打开成本较高。

因此本次将“复杂表单使用抽屉”提升为仓库级强制规则，并同步收敛现有页面。

## 2. 目标

1. 为复杂表单提供更稳定的横向空间和更清晰的分区结构。
2. 保留用户对列表页、统计卡片和背景上下文的感知，减少跳出感。
3. 给页面评审、自检和后续开发提供统一判定标准。

## 3. 判定规则

### 3.1 必须使用抽屉的场景

- 字段数量较多，或明显超过一个短表单的心智负担。
- 表单分成多个区域、步骤、卡片或配置块。
- 包含 `Form.List`、动态条件、级别块、调试参数、长文本、JSON、脚本、表达式等复杂配置。
- 包含较强的上下文依赖，需要边看列表边配置或反复比对。
- 包含预览、调试结果、运行结果、条件说明等伴随型信息。

### 3.2 可以保留弹窗的场景

- 简短创建或编辑，仅包含少量基础字段。
- 确认、提示、二次校验。
- 只读详情、图片预览、日志详情、小范围状态调整。

## 4. 本次收敛范围

本次已将下列明显属于复杂表单的页面改为抽屉：

- 告警规则维护
- 规则引擎维护
- 定时任务维护
- 租户维护中的创建、授权、配额配置
- 产品维护
- 设备批量注册
- 协议解析器编辑与上下行调试
- Modbus 采集任务注册
- SNMP 采集任务注册
- 消息模板维护

保留弹窗的页面，当前主要属于确认、详情、小表单或短流程交互。

## 5. 设计取舍

### 5.1 为什么不是所有表单都改成抽屉

抽屉适合复杂配置，但对于只有少量字段的表单，弹窗仍然更轻、更快，视觉负担也更小。统一把所有交互都改成抽屉，会让轻量操作反而变重。

### 5.2 为什么不继续通过“加宽弹窗”解决

加宽弹窗可以暂时缓解拥挤问题，但无法解决复杂表单的上下文保留、区域分层和长流程可编辑性问题。抽屉更适合承载配置型页面。

## 6. 后续约束

1. 新增前端页面时，创建、编辑、调试、注册、配置类表单必须先做复杂度判断。
2. 评审时需要明确说明为什么使用抽屉或为什么保留弹窗。
3. 如页面后续从简单表单演进成复杂表单，必须同步把容器从弹窗迁移为抽屉。
