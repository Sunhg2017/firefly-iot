# 管理员会话页重构设计

## 1. 背景

系统运维空间的 `安全管理 -> 管理员会话` 原实现要求手工输入 `userId` 后才能查询某个用户的会话。

这带来了三个直接问题：

- 页面默认没有任何管理员会话，看起来像“明明在线却没有数据”
- 交互依赖数据库主键，不符合仓库“优先使用业务唯一键、不要把主键暴露给前端用户”的规则
- 前端会话表字段与后端返回不一致，管理员会话和我的会话中的 IP、会话信息展示不完整

同时，该页面挂在系统运维空间，真实诉求并不是“按 ID 查某个人”，而是“平台侧查看当前在线的管理员会话”。

## 2. 目标

- 把 `管理员会话` 收口为默认可见的在线管理员会话列表
- 平台侧默认展示当前在线的系统运维账号和各租户超级管理员
- 去掉手输 `userId` 的旧交互，改为列表 + 筛选 + 批量操作
- 保持前后端字段一致，补齐平台、登录方式、IP、User-Agent、活跃时间等信息

## 3. 范围

涉及模块：

- `firefly-system`
- `firefly-web`
- `docs/design`
- `docs/operations`
- `docs/user-guide`

不涉及：

- 菜单、权限点和所属空间调整
- Flyway 菜单台账 SQL 变更
- 历史会话数据迁移

## 4. 方案

### 4.1 管理员口径

管理员会话按以下口径收敛：

- `SYSTEM_OPS` 用户：视为系统运维管理员
- `tenants.admin_user_id = users.id` 的用户：视为租户超级管理员

当前实现不再继续保留“按任意用户 ID 查询会话”的旧模式。

### 4.2 后端接口

移除旧的按用户 ID 查询接口，改为管理员会话专用接口：

- `POST /api/v1/admin-sessions/list`
  - 分页查询在线管理员会话
  - 支持 `keyword`、`platform`、`adminType` 筛选
- `DELETE /api/v1/admin-sessions/{sessionId}`
  - 强制下线单个管理员会话
- `POST /api/v1/admin-sessions/users/{username}/kick`
  - 按管理员账号强制下线其全部在线会话

### 4.3 查询范围

- 当前用户是系统运维账号时：可查看全平台在线管理员会话
- 当前用户不是系统运维账号时：查询范围收敛到当前租户，避免跨租户读取

为支持系统运维空间跨租户查看管理员会话，`UserSessionMapper` 新增忽略租户线的专用查询方法，再由服务层按当前操作者身份补充范围约束。

### 4.4 前端交互

`SecurityPage` 中的 `管理员会话` 页签改为：

- 默认展示在线管理员会话列表
- 支持关键字搜索：
  - 管理员账号
  - 姓名
  - 租户名称
  - 登录 IP
- 支持平台筛选
- 支持管理员身份筛选
- 支持单会话下线
- 支持按管理员账号全部下线

同时顺手修正：

- `我的会话` 改为读取后端真实字段 `loginMethod`、`loginIp`
- `管理员会话` 也统一使用真实字段，不再读不存在的 `sessionId`、`ip`

## 5. 数据结构

新增后端视图对象：

- `AdminSessionQueryDTO`
- `AdminSessionVO`
- `AdminSessionTarget`

`AdminSessionVO` 主要包含：

- `id`
- `username`
- `realName`
- `tenantName`
- `adminType`
- `platform`
- `loginMethod`
- `loginIp`
- `userAgent`
- `lastActiveAt`
- `createdAt`

## 6. 关键取舍

- 没有继续兼容旧的 `/api/v1/users/{userId}/sessions` 和 `/api/v1/users/{userId}/kick`，而是直接收口到新的管理员会话接口
- 页面操作主入口改为管理员账号、租户名等业务字段，不再要求用户记忆数据库主键
- “管理员”当前按系统运维账号与租户超级管理员定义，不额外引入基于角色名模糊匹配的口径，避免误判

## 7. 风险

- 如果未来需要把“管理员”扩展到更多角色，需要在后端统一补充管理员识别规则，而不是继续回退到手输用户 ID
- 当前页签在系统运维空间中使用最合理；若后续在租户空间复用，查询范围会自动收敛到当前租户
