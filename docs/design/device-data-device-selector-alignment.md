# 设备数据页设备选择收口设计

## 背景

`/device-data` 页面原先仍要求用户手工输入数据库 `deviceId` 才能查询最新值、历史数据和聚合统计；设备事件列表也直接展示 `deviceId`。

这与仓库内“优先使用业务唯一键，不把数据库主键暴露给前端用户”的规则冲突，也让联调时必须额外查一次内部 ID。

## 目标

- 设备数据页统一改为按 `deviceName` 口径选择设备。
- 保留后端现有按 `deviceId` 查询的实现，但把主键收敛在页面内部传值层。
- 设备事件列表补齐 `deviceName`，不再把数据库主键作为主视角信息展示。

## 范围

本次仅覆盖：

- `firefly-web` 的 `设备数据` 页面
- `firefly-device` 的设备事件列表返回体补齐 `deviceName`

不涉及：

- 设备数据查询 SQL 结构调整
- 新增设备筛选接口
- 菜单、权限或路由变更

## 方案

### 1. 页面查询入口改为设备选择器

`最新数据 / 历史数据 / 聚合统计 / 设备事件` 四个页签统一复用同一套设备搜索下拉：

- 通过 `deviceApi.list` 按关键词加载设备
- 下拉展示 `nickname + deviceName + 在线/状态`
- 页面展示和筛选使用 `deviceName` 语义
- 真正发请求时仍使用选中项内部携带的 `deviceId`

### 2. 搜索结果保留已选设备

设备搜索是远端分页查询，若用户已经选中过设备，再输入新关键词时不能让旧选项从 Select 中消失，否则控件会退化成只显示内部数值。

因此前端改为“增量合并选项集”：

- 新结果覆盖同 `deviceId` 的旧选项
- 已选设备即使不在当前搜索结果中，也继续保留在本地选项集中

### 3. 设备事件返回体补齐业务标识

`DeviceEventVO` 新增：

- `deviceName`

`DeviceDataService.listEvents(...)` 在分页结果返回前按当前页 `deviceId` 批量回查设备表，补齐 `deviceName` 后再返回给前端。

这样设备事件列表可以直接展示业务标识，不再依赖页面额外拼装数据库主键。

## 接口变化

### 设备事件列表

`POST /api/v1/device-events/list`

响应项新增：

- `deviceName: string`

现有字段保持不变，前端可逐步从 `deviceId` 展示切到 `deviceName` 展示。

## 风险与取舍

- 本次没有新增专门的“按业务键查询设备数据”后端接口，仍沿用现有 `deviceId` 查询能力，降低了改动面。
- 设备事件列表补齐 `deviceName` 只针对当前页结果批量查询，避免为列表增加额外联表 SQL。
- 若旧数据对应的设备已被删除，事件列表会回退为“设备信息缺失”，而不是重新展示内部主键。
