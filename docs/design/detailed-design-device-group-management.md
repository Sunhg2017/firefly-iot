# Firefly-IoT 设备分组模块详细设计

> 版本: v1.0.0
> 日期: 2026-03-13
> 状态: Done

## 1. 背景

设备分组原有实现存在以下问题：

- 页面把分组树、分组详情、成员维护堆叠在一起，信息层次不清晰。
- 静态分组成员维护依赖手工输入设备 ID，违反“优先选择型控件、避免用户手填内部主键”的仓库规则。
- 编辑分组时页面允许维护 `type` 和 `parentId`，但后端更新接口未完整落库，前后端行为不一致。
- 动态分组与静态分组共用成员维护入口，容易误导用户手工维护本应由规则归集的成员。
- 分组详情仍可能展示父分组 ID、设备 ID 等内部主键，不利于租户侧使用。

本次优化目标是把设备分组收敛为“目录浏览 + 画像查看 + 成员维护”的清晰交互，并同步补齐后端约束，保证前后端行为一致。

## 2. 范围

本次变更覆盖：

- 前端页面 `firefly-web/src/pages/device-group/DeviceGroupPage.tsx`
- 设备分组更新接口与成员明细接口
- 设备分组服务中的父子层级校验、静动态分组成员约束、成员业务字段补齐

本次不包含：

- 动态分组规则 DSL 升级
- 批量导入导出
- 分组授权模型调整

## 3. 设计目标

- 用户先看分组全局，再选目录节点，再看当前分组画像和成员。
- 设备加入静态分组时只允许通过设备名称/昵称搜索选择，不再让用户输入设备 ID。
- 动态分组明确为“规则驱动、成员只读”模型。
- 分组编辑时 `type`、`parentId`、`dynamicRule` 的保存逻辑与页面保持一致。
- 前端主视角不再展示数据库主键，改为展示分组名称、设备名称、产品名称、`productKey` 等业务字段。

## 4. 前端交互设计

### 4.1 页面结构

设备分组页拆分为三层：

1. 页面总览
   - 展示分组总数、静态分组数、动态分组数、静态成员总量。
2. 分组目录
   - 左侧树结构展示分组层级。
   - 支持名称搜索，搜索结果保留祖先路径，便于定位。
3. 分组工作区
   - 分组画像：展示名称、描述、类型、成员数、创建时间、父分组名称。
   - 分组成员：静态分组显示成员表格和“加入设备”选择器；动态分组显示只读说明。

### 4.2 分组编辑

- 继续使用弹窗而不是抽屉，因为当前表单字段较少，属于简单表单。
- `parentId` 使用下拉选择，选项名称为分组名，不向用户暴露主键。
- 编辑时自动排除当前分组及其子孙节点，避免用户在前端选出非法父分组。
- 仅当类型为 `DYNAMIC` 时展示 `dynamicRule` 输入框。

### 4.3 成员维护

- 仅静态分组显示“加入设备”入口。
- “加入设备”使用远程搜索下拉框，从设备列表接口查询候选设备。
- 候选项显示设备名称和昵称，并自动过滤已经加入当前分组的设备。
- 成员表展示：
  - 设备名称
  - 昵称
  - 产品名称
  - `productKey`
  - 设备状态
  - 在线状态
  - 加入时间

## 5. 后端设计

### 5.1 更新分组

`DeviceGroupController#updateGroup` 改为透传：

- `name`
- `description`
- `type`
- `dynamicRule`
- `parentId`

`DeviceGroupService#updateGroup` 负责：

- 归一化分组类型，只允许 `STATIC`、`DYNAMIC`
- 归一化父分组，`null` 或 `<= 0` 视为顶级分组
- 校验父分组是否存在且属于当前租户
- 校验父分组不能选择自己或自己的子孙节点
- 动态分组才保存 `dynamicRule`，静态分组自动清空规则

### 5.2 成员明细

新增 `listMemberDetails`，在原始成员关系基础上补齐：

- `deviceName`
- `nickname`
- `productName`
- `productKey`
- `status`
- `onlineStatus`

这样前端可以直接展示业务字段，不再以设备 ID 作为主显示信息。

### 5.3 静动态分组约束

服务层新增统一限制：

- 动态分组不允许手工添加成员
- 动态分组不允许手工移除成员
- 动态分组不允许批量维护成员
- 静态分组添加成员前必须校验设备存在且未删除

## 6. 关键流程

### 6.1 新建静态分组

1. 用户点击“新建分组”或“新建子分组”
2. 页面填写名称、上级分组、类型、描述
3. 提交后后端归一化父分组和类型
4. 成功后刷新分组树并重新计算总览统计

### 6.2 新建动态分组

1. 用户选择分组类型为 `DYNAMIC`
2. 页面显示动态规则输入框
3. 后端保存规则内容
4. 页面成员区域切换为只读提示，不提供手工加减成员入口

### 6.3 静态分组加入设备

1. 页面根据关键字调用设备列表接口搜索设备
2. 用户从下拉选中设备
3. 后端校验分组类型和设备存在性
4. 保存成员关系后刷新成员列表和分组树统计

## 7. 关键设计取舍

- 没有兼容旧的“手输设备 ID”交互，因为该方式本身不合理，且用户确认系统尚未形成历史配置依赖。
- 动态分组仍暂时保留文本形式规则输入，原因是本次重点在于分组体验和行为校正，不在规则 DSL 升级。
- 父分组选择同时在前端和后端做限制，前端用于减少误操作，后端用于兜底保证数据正确性。

## 8. 风险与后续建议

- 动态分组规则目前仍是文本输入，后续可升级为结构化规则编辑器。
- 设备搜索依赖 `/devices/list` 返回的设备名称、昵称和状态字段，后续若接口裁剪字段需要同步验证页面。
- 若后续增加分组规模统计口径，需要区分“静态成员数”和“动态命中数”，避免混用当前 `deviceCount`。
