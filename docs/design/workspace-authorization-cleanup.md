# 工作空间授权与菜单收口设计

## 1. 背景

租户空间菜单、角色可分配权限和前端导航应当共享同一套授权边界。系统未正式使用前，不再为旧实现保留“默认菜单”“默认权限”之类的过渡逻辑。

本次修复针对一个严重授权缺陷：新租户在系统运维尚未分配任何租户空间菜单时，首次登录仍能看到全部租户空间菜单。根因是租户创建阶段自动写入了全量 `tenant_menu_configs` 记录。

## 2. 目标

- 租户空间菜单必须完全由系统运维显式授权决定。
- 新租户创建后默认不授予全部租户空间菜单。
- 角色可分配权限、前端菜单显示、首页跳转结果保持一致。
- 删除默认全量授权代码路径，避免后续再次被误用。

## 3. 方案

### 3.1 删除创建租户时的默认菜单授权

- `TenantService.createTenant(...)` 不再调用租户空间全量默认授权逻辑。
- 租户初始化仅保留租户、管理员用户与管理员角色创建。

### 3.2 收口授权来源

- `tenant_menu_configs` 仅表示系统运维对租户空间菜单的显式分配结果。
- `WorkspaceMenuAccessService` 通过 `TenantWorkspaceMenuService.listAuthorizedRoutePaths(...)` 计算当前用户可访问菜单路径。
- `WorkspacePermissionCatalogService.getTenantWorkspacePermissions(...)` 只返回当前租户已授权菜单对应的权限点。

### 3.3 删除冗余默认授权接口

- 删除 `TenantMenuConfigService.grantDefaultMenus(...)`。
- 删除 `TenantWorkspaceMenuService.grantDefaultMenus(...)`。
- 后续如需初始化租户菜单，统一走显式的“更新租户空间授权”接口，而不是重新引入默认全量授权。

## 4. 运维提醒

- 历史环境若已经存在由旧逻辑生成的全量 `tenant_menu_configs` 数据，需要按租户实际授权要求清理。
- 清理后由系统运维重新在租户管理页面勾选需要开放的租户空间菜单。

## 5. 验证要点

- 新建租户后，在未配置空间授权前，租户管理员登录不应看到全部租户业务菜单。
- 系统运维完成空间授权后，租户重新登录应只看到被授权菜单。
- 角色管理页面中可分配权限应与当前已授权租户菜单保持一致。
