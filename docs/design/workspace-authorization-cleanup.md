# 工作空间授权与菜单收口设计

## 1. 背景

系统运维空间与租户业务空间此前混入了多套过渡逻辑：

- 租户侧自行维护菜单树。
- 通过系统设置维护“租户管理员默认权限”。
- 登录态里保留 `workspaceMenuAdmin` 之类的过渡字段。
- 协议解析发布策略对 `DEVICE_LIST` 同时兼容 `deviceIds` 与 `deviceNames`。

这些实现会把权限来源、菜单来源和页面可见性拆成多条链路，导致前后端都要做兜底判断，持续放大维护成本。

## 2. 目标

- 平台空间和租户空间都具备独立的用户、角色、权限管理能力。
- 租户空间可用菜单只由系统运维在租户管理中授权，不再允许租户侧自维护菜单。
- 角色可分配权限只来自当前工作空间的权限目录，不再依赖“默认权限配置”二次兜底。
- 协议解析发布策略收口为 `deviceNames`，不再保留旧的 `deviceIds` 兼容路径。
- 前端导航、首页跳转、403 回退统一基于路由目录和当前权限集计算。

## 3. 设计方案

### 3.1 权限与菜单的唯一来源

- `routes.tsx` 作为前端菜单与路由的唯一目录。
- `WorkspacePermissionCatalogService` 作为后端权限目录与可授权模块的唯一来源。
- `tenant_menu_configs` 只保留“平台给租户分配空间功能”的持久化职责，不再提供租户自助增删改接口。

### 3.2 工作空间职责划分

- 系统运维空间：
  - 管理租户。
  - 管理系统运维空间的用户与角色。
  - 给租户授权其可使用的租户空间功能。
  - 维护系统级通知渠道、系统设置、监控审计等平台能力。
- 租户业务空间：
  - 管理本租户的用户与角色。
  - 角色只能分配已获授权的租户空间权限。
  - 使用业务菜单，不再存在“租户管理员默认权限配置”与“租户菜单自维护”入口。

### 3.3 租户管理员与角色权限收口

- 新租户创建时，租户管理员角色直接同步当前租户已授权的全部租户空间权限。
- 系统运维调整租户空间授权后，系统重新收敛租户下各角色权限：
  - 系统内置租户管理员角色同步到当前全部已授权权限。
  - 其他租户角色保留与当前授权交集，超出授权范围的权限自动移除。
- `RoleController` 新增角色选项和权限分组选项接口，前端按当前空间拉取可选项，不再自行拼接或猜测权限。

### 3.4 前端导航收口

- `BasicLayout` 直接根据路由目录与权限集渲染侧边栏。
- `useAuthStore` 只缓存用户信息和权限，不再缓存租户菜单配置或过渡字段。
- 登录后首页、403 跳转、工作空间切换统一走 `workspaceRoutes.ts` 的目录计算结果。

### 3.5 协议解析发布策略收口

- `DEVICE_LIST` 策略仅允许维护 `deviceNames`。
- 发布规则摘要、JSON 提示、匹配逻辑全部同步到 `deviceNames` 口径。
- 下行调试和规则维护继续坚持业务标识优先，不再对外暴露内部主键兼容路径。

## 4. 移除内容

- 后端删除：
  - `/api/v1/tenant/menu-configs/*`
  - “租户管理员默认权限”相关 DTO、Service、Controller
  - `workspaceMenuAdmin` 相关登录与用户返回字段
- 前端删除：
  - 租户菜单自维护页面
  - 依赖 `menuConfig` 的登录跳转、布局渲染与权限兜底
  - 协议解析发布策略中的 `deviceIds` 页面提示与摘要

## 5. 关键取舍

- 不再为旧菜单、旧权限、旧发布配置保留双轨逻辑，因为系统尚未正式启用，最佳实现优先于兼容。
- 若历史数据库里仍残留旧配置，应通过数据清理解决，而不是继续在业务代码中保留过渡判断。

## 6. 数据清理建议

建议上线前清理以下旧数据，避免误导后续运维：

- `system_configs` 中 `tenant.admin.default-permissions`
- 不符合当前租户空间授权结果的历史 `tenant_menu_configs`
- 协议解析规则发布配置中残留的 `deviceIds`

## 7. 验证要点

- 平台管理员登录后可见系统运维菜单、用户、角色、租户管理。
- 租户管理员登录后可见租户业务菜单、用户管理、角色管理。
- 调整租户空间授权后，租户角色权限会被收敛到授权范围内。
- 协议解析页面保存 `DEVICE_LIST` 规则时仅生成 `deviceNames`。
