# 前端根路由首页跳转修复设计
## 1. 背景

`firefly-web` 根路由 `/` 原先固定跳转到 `/dashboard`。当当前用户没有仪表盘菜单或 `dashboard:read` 权限时，布局层会继续按照已授权菜单和权限做拦截，最终把用户送到 `/403`。

这会导致一个明显的体验问题：用户实际上拥有其他业务页面访问权，但只因为根路由写死到了一个自己无权访问的页面，就会在访问系统根目录时直接看到 403。

## 2. 目标

- 根路由 `/` 必须跳转到当前用户真实可访问的首页，而不是写死到某个菜单。
- 站内未知路由的兜底逻辑必须与登录后首页、403 返回首页保持一致。
- 保持现有菜单权限模型不变，不新增兼容分支，不回退到旧菜单或旧权限口径。

## 3. 范围

本次仅调整前端首页解析与路由跳转行为，涉及：

- `firefly-web/src/config/workspaceRoutes.ts`
- `firefly-web/src/App.tsx`
- `firefly-web/src/layouts/BasicLayout.tsx`
- `firefly-web/src/pages/login/index.tsx`
- `firefly-web/src/pages/403.tsx`

本次不涉及：

- 菜单结构变更
- 权限点变更
- `workspace_menu_permission_catalog` 台账变更
- 后端接口协议变更

## 4. 方案

### 4.1 首页解析统一收口

在 `workspaceRoutes.ts` 中新增 `getUserHomePath(user)`，统一根据以下信息计算首页：

- `userType` 对应的平台空间/租户空间
- `permissions`
- `authorizedMenuPaths`

最终仍复用现有 `getWorkspaceHomePath(...)` 逻辑，保证首页解析口径只有一处。

### 4.2 根路由与站内兜底改为动态跳转

在 `App.tsx` 中新增 `HomeRedirect` 组件，直接读取当前登录用户并跳转到 `getUserHomePath(user)` 结果：

- `/` 不再固定跳转 `/dashboard`
- 站内 `*` 兜底也不再固定跳转 `/dashboard`

这样可以保证无论是直接访问根路径，还是访问一个不存在的站内路径，都会回到当前用户第一个可访问页面。

### 4.3 登录页与 403 页复用同一算法

登录成功后的首页跳转、403 页“返回首页”按钮也改为复用 `getUserHomePath(user)`，避免三套实现发生偏差。

### 4.4 布局层放行根路径与站内兜底跳转

`BasicLayout` 本身也会对当前路径执行一次权限与菜单可访问性校验。如果这里直接把 `/` 或未注册的站内路径判成不可访问，就会在 `HomeRedirect` 运行前先跳到 `/403`，导致根路径修复失效。

因此布局层需要额外遵守两条规则：

- 当前路径为 `/` 时，先放行，让索引路由的 `HomeRedirect` 完成首页解析。
- 当前路径不在已注册路由表中时，先放行，让站内 `*` 兜底路由的 `HomeRedirect` 完成回首页逻辑。

## 5. 关键流程

1. 用户访问 `/`
2. `PrivateRoute` 确保已登录且用户信息已加载
3. `BasicLayout` 放行 `/`，不提前送 `/403`
4. `HomeRedirect` 调用 `getUserHomePath(user)`
5. 若存在可访问页面，则跳转到该页面
6. 若当前用户确实没有任何可访问页面，则跳转到 `/403`

## 6. 风险与取舍

- 本次没有新增后端兜底逻辑，因为问题本质在前端路由层，直接在当前最佳实现收口即可。
- 如果某个用户所有 `authorizedMenuPaths` 都为空，首页仍会落到 `/403`。这是符合授权结果的表现，不应再通过代码偷偷放开默认菜单。
- 由于未改动菜单定义、权限点和空间归属，因此不需要新增 Flyway SQL 台账记录。
- 根路径和未知路径只是在跳转发生前被布局层短暂放行；真正已注册且无权访问的业务路径仍然保持 `/403`，不会削弱现有授权约束。

## 7. 验证要点

- 有仪表盘权限的用户访问 `/`，仍可进入自己的默认首页。
- 无仪表盘权限但有其他菜单权限的用户访问 `/`，应进入首个可访问业务页面而不是 `/403`。
- 登录成功跳转、403 返回首页、未知站内路径兜底三者行为一致。
- 已注册但未授权的业务页面仍然进入 `/403`，不应被本次修复误放开。
