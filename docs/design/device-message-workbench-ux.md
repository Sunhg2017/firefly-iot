# 设备消息工作台体验优化设计

## 背景

原设备消息页是三个简单表单的并列堆叠：

- 每个标签页都要重复选择一次设备
- 属性设置、服务调用、原始消息都依赖手工拼 JSON
- 用户看不到当前设备上下文、物模型能力摘要和最近一次发送结果

这导致页面更像内部调试入口，而不是可持续操作的消息下发工作台。

## 目标

- 把设备选择提升为页面级共享上下文，减少重复操作
- 基于产品物模型生成属性与服务参数模板，降低手工输入成本
- 在页面内直接呈现设备状态、产品信息、可写属性数、可调服务数和最近一次发送结果
- 保留原有三类下发能力，但把操作流程收敛为“选设备、选场景、改模板、发送”

## 方案

## 1. 页面顶部设备上下文区

- 顶部统一提供设备搜索选择框
- 右侧展示设备标识、展示名称、所属产品、在线状态、设备状态
- 同步展示当前产品的可写属性数量和可调服务数量
- 最近一次发送成功或失败后，在顶部保留结果反馈，便于连续调试

## 2. 属性设置工作区

- 使用 `CodeEditorField` 代替普通文本框编辑属性 JSON
- 从物模型中提取可写属性，支持“填充已选属性”和“填充全部可写属性”
- 右侧展示当前产品的可写属性列表
- 发送时继续调用现有 `setProperty` 接口，不改后端协议

## 3. 服务调用工作区

- 仅展示物模型中的真实自定义服务
- 服务切换后自动带入输入参数模板，但尽量保留用户已改过的草稿
- 右侧展示服务标识、名称、描述、输入输出参数数量
- 通过标签展示输入参数摘要，减少用户来回查看物模型

## 4. 原始消息工作区

- 按消息类型提供示例模板
- 切换类型时尽量自动套用该类型的示例 JSON
- 明确标识这是高级调试入口，避免与标准属性设置、服务调用混淆

## 设计取舍

- 不新增后端接口，继续复用现有设备消息 API
- 不做旧版简陋表单与新版工作台双轨兼容，页面直接收口到新的单设备工作台
- 不把所有物模型字段强行自动回填到编辑器，避免覆盖用户已修改的草稿，只在仍是自动模板时更新

## 风险与验证点

- 需要验证频繁切换设备或服务时，编辑器草稿不会被意外覆盖
- 需要验证无可写属性、无服务、物模型加载失败等空态是否清晰
- 需要验证顶部结果反馈和下方工作区不会在小屏幕上相互挤压
