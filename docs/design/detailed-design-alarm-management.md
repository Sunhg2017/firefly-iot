# Firefly IoT 告警管理详细设计

## 1. 背景

告警模块已经拆分为两个菜单：

- 告警规则维护
- 告警处理

其中“告警规则维护”面向规则管理员。虽然规则协议已经改成结构化 JSON，但页面仍然容易让用户陷入两类困惑：

1. 不清楚应该先定级别还是先写条件。
2. 一长串表单字段缺少步骤感，维护多级别规则时很难快速判断当前在配置哪一层。

本次进一步收口页面交互，统一采用下面的思路：

1. 一个规则包含多个“级别块”。
2. 每个级别块先选告警级别。
3. 再选该块的触发语义。
4. 最后维护该块下的条件。

## 2. 目标与范围

### 2.1 目标

1. 页面交互更直观，用户能明显感知“配置顺序”。
2. 每个级别块清晰表达“级别 + 触发语义 + 条件列表”的层次。
3. 保持结构化规则协议不变，继续使用 `version = 3`。
4. 不引入数据库迁移。

### 2.2 范围

- `firefly-web` 告警规则维护表单交互优化
- `firefly-rule` 继续沿用当前结构化规则校验
- 告警模块三类文档更新

## 3. 协议与数据模型

数据库结构不变：

- `alarm_rules.condition_expr` 保存完整规则 JSON
- `alarm_rules.level` 保存自动推导出的规则主级别

结构化规则仍为：

```json
{
  "mode": "STRUCTURED",
  "version": 3,
  "groups": [
    {
      "level": "CRITICAL",
      "triggerMode": "ALL",
      "conditions": [
        {
          "type": "THRESHOLD",
          "metricKey": "temperature",
          "aggregateType": "LATEST",
          "operator": "GT",
          "threshold": 85
        }
      ]
    }
  ]
}
```

每个级别块包含：

- `level`
- `triggerMode`
- `matchCount`
- `conditions[]`

## 4. 前端交互设计

### 4.1 页面结构

规则配置抽屉调整为五个明显区域：

1. 配置步骤总览
2. 基本信息
3. 适用范围与主级别
4. 告警级别块列表
5. 整体规则预览

该页面同时作为仓库级“复杂表单使用抽屉”规则的典型落地案例。告警规则维护包含多级别块、触发语义、条件列表、实时预览和较长配置流程，已明确归类为复杂表单，不再使用居中弹窗。

### 4.2 引导式交互

顶部增加“配置步骤”区块，用于告诉用户当前页面推荐顺序：

1. 先填规则名称和说明
2. 再选项目/产品/设备范围
3. 最后配置级别块

同时展示实时统计：

- 已配置级别块数
- 已配置条件数
- 当前规则主级别

这样用户在长表单中也能一直看到当前配置规模和结果。

### 4.3 级别块卡片化

每个级别块使用独立卡片展示，并按告警级别附加不同强调色。

卡片顶部直接展示：

- 级别块名称
- 告警级别标签
- 触发语义标签
- 条件数量标签

卡片内部继续拆成三个步骤面板：

1. 选择告警级别
2. 选择触发语义
3. 明确触发策略

然后再进入条件配置区。

### 4.4 选择控件优化

以下枚举型字段由下拉改为更直观的分段选择：

- 告警级别
- 触发语义
- 条件类型

这样用户不需要反复打开下拉框比较选项，更适合连续配置多个级别块。

### 4.5 条件编辑器优化

条件卡片中新增两类信息：

1. 条件类型说明
2. 条件预览

其中：

- 选择条件类型后，会立即显示适用场景说明
- 条件参数单独放入“条件参数”卡片
- 底部实时生成自然语言预览

如果尚未选择产品且当前条件不是自定义表达式，页面会提醒“先选产品再选指标”，减少手工输入。

### 4.6 预览机制

页面保留两层预览：

1. 单条件预览
2. 单级别块预览
3. 全规则预览

这样用户可以从局部到整体逐层确认。

## 5. 后端设计

本次不调整规则协议，只继续复用现有校验逻辑：

1. 根节点必须是 `mode = STRUCTURED`
2. 必须包含非空 `groups[]`
3. 每个级别块都必须有合法 `level` 和 `triggerMode`
4. `AT_LEAST` 必须校验 `matchCount`
5. 每条条件按类型校验必填字段
6. 规则主级别按最高严重级别自动推导

## 6. 关键流程

### 6.1 新建规则

1. 用户先看顶部配置步骤和汇总信息
2. 维护基本信息
3. 选择适用范围
4. 新增一个级别块
5. 在级别块内完成“级别 -> 触发语义 -> 条件”配置
6. 页面实时生成预览
7. 提交后保存结构化 JSON

### 6.2 编辑规则

1. 打开规则
2. 反序列化 `conditionExpr`
3. 回填为多个级别块卡片
4. 用户按级别块逐个调整

## 7. 设计取舍

### 7.1 为什么强调步骤感

因为规则本身已经足够复杂，多级别、多条件、多触发语义如果全部平铺在一个区域中，用户很难建立“正在配置哪一层”的心智模型。

### 7.2 为什么使用卡片和分段选择

原因有三个：

1. 层级更清晰
2. 枚举选项更容易比较
3. 更适合连续批量维护多个级别块

### 7.3 为什么保留多层预览

因为规则越复杂，用户越需要快速确认：

- 单条条件写对没有
- 这个级别块语义对不对
- 整条规则最终表达的告警策略是什么

## 8. 风险与后续

### 8.1 风险

1. 下游如果仍按旧结构消费规则，会无法识别当前协议。
2. 产品物模型不完整时，指标选择体验会下降。

### 8.2 后续建议

1. 增加规则模板能力，减少重复配置。
2. 增加条件排序/复制功能，进一步提高配置效率。
3. 在执行链路中补齐 `groups[]` 的命中计算与调试能力。
