# 设备标签与设备主流程联动设计

## 背景

原有实现同时存在两套标签模型：

- `devices.tags` 文本/JSON 字段
- `device_tags`、`device_tag_bindings` 独立标签表

设备管理页仍要求手工输入标签文本，设备标签页又要求手工输入 `deviceId` 进行绑定，导致标签定义、绑定、展示三段链路脱节。

## 目标

- 设备创建、编辑、批量导入直接使用结构化标签选择
- 设备列表直接展示绑定标签
- 设备标签页绑定设备时从现有设备选择，不再手工输入 `deviceId`
- 后端统一以 `device_tag_bindings` 为真实来源

## 范围

- `firefly-device`
  - 设备 DTO 增加 `tagIds`
  - 设备详情/列表返回 `tagList`
  - 设备标签服务补齐绑定校验、批量同步、删除清理、快照刷新
- `firefly-web`
  - 设备管理页接入标签选择与展示
  - 设备标签页改为设备选择绑定

## 核心设计

### 1. 标签真实来源统一为绑定表

- 设备和标签的关系统一从 `device_tag_bindings` 读取
- `devices.tags` 不再作为前台配置入口
- `devices.tags` 仅保留为快照字段，内容为 JSON 数组，例如：

```json
["region:cn-east", "level:critical"]
```

### 2. 设备表单统一提交 `tagIds`

- `DeviceCreateDTO`
- `DeviceUpdateDTO`
- `DeviceBatchCreateDTO`
- `DeviceImportDTO`

以上 DTO 均改为使用 `tagIds`，避免自由文本标签继续进入系统。

### 3. 服务层新增同步逻辑

`DeviceTagService.syncDeviceTags(deviceId, tagIds)` 负责：

- 校验设备是否存在且属于当前租户
- 校验标签是否存在且属于当前租户
- 对比当前绑定关系，增删差异项
- 回刷标签 `device_count`
- 刷新设备标签快照

### 4. 删除与批量场景统一收口

- 删除设备时先清理标签绑定
- 删除标签时刷新受影响设备的快照
- 批量绑定/解绑标签后同步刷新设备快照

## 接口变化

### 设备接口

- 创建设备：支持 `tagIds`
- 编辑设备：支持 `tagIds`
- 批量导入设备：支持 `tagIds`
- 设备列表/详情：返回 `tagList`

### 标签接口

- 查询标签绑定设备：返回设备名称、别名、产品名称等可展示信息

## 关键取舍

- 没有继续保留前端“文本标签”兼容入口
  - 原因：当前系统实际未形成有效配置资产，继续兼容会让双模型长期并存
- 没有新增设备标签专用选择接口
  - 当前设备数量中等时，使用现有分页查询接口做远程搜索即可满足页面需求

## 风险与控制

- 风险：绑定表与快照字段不一致
  - 控制：所有绑定写操作统一收口到 `DeviceTagService`
- 风险：跨租户误绑定
  - 控制：设备、标签绑定前都做租户校验
