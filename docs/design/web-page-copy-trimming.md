# 前端页面说明文案收口设计

## 1. 背景

前端多个页面长期叠加了页头说明、提示卡片、步骤说明和辅助文案。部分内容不直接服务当前操作，占用页面空间，也与仓库“页面只保留当前操作直接有帮助文案”的规则不一致。

## 2. 目标

- 收口页面头部和工作区中的冗余说明。
- 保留结果反馈、风险提示、字段约束和直接影响操作成败的信息。
- 将背景性、理念性、跨页面流程性说明转移到 `docs/` 文档，而不是继续留在页面里。

## 3. 范围

本次覆盖以下前端区域：

- 协议接入类页面：`Modbus`、`SNMP`、`TCP/UDP`、`WebSocket`、`LoRaWAN`
- OpenAPI 与 AppKey 页面
- 用户、角色、菜单、权限页面
- 设备影子、设备消息、设备标识页面
- 工作台、数据分析、设备数据、通知、产品、租户、规则、消息模板、OTA 等列表页头
- 设备分组、设备拓扑等选择型页面

## 4. 收口规则

### 4.1 保留

仅保留下列文案：

- 保存、删除、清空、发布等操作的影响说明
- 无法从控件本身看出的字段约束
- 运行结果、错误反馈、校验失败原因
- 与当前所选对象直接相关的动态信息

### 4.2 删除或压缩

优先删除以下内容：

- 页头“先做什么、再做什么”式说明
- 与当前控件无关的背景介绍
- 可以从页面结构直接看出的重复引导
- 大段“说明 / 使用规则 / 操作提示 / 编辑说明”提示卡片
- 全屏编辑、步骤切换、只读区等显而易见行为的重复解释

## 5. 实现策略

- `PageHeader` 的说明仅在承载统计口径或动态上下文时保留；纯操作引导直接删除。
- 协议接入页统一将 `ProtocolAccessGuide` 收口为精简的相关入口卡片，只保留入口按钮和端点信息。
- 设备影子、设备消息等复杂工作区优先保留状态、结果、错误和必要约束，移除解释性提示。
- 页面仍需解释的行为统一转移到设计、运维、使用文档中维护。

## 6. 风险与取舍

- 删除过多提示可能增加首次使用者摸索成本，因此保留了操作结果、危险动作、字段格式等直接影响成功率的信息。
- 本次不改接口、不改权限、不改业务流程，只调整页面可见文案与部分提示卡片结构。
