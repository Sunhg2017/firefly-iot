# 告警通知编排与接收组设计

## 1. 背景

现有告警规则只能维护触发条件，无法直接表达“通过什么方式通知谁”。这会导致：

- 告警规则维护人与通知通道维护人之间需要线下约定。
- 告警接收对象只能靠临时口头约定，无法形成可复用的值班组。
- 规则配置页面难以形成完整闭环，规则触发后是否会送达无法直观看出。

本次改造的目标是把告警规则中的“通知方式”和“接收对象”结构化，同时补齐租户级的“告警接收组”能力。

## 2. 目标与范围

### 2.1 目标

- 告警规则支持维护通知方式。
- 告警规则支持按接收组、按指定人员、或二者混合配置接收对象。
- 告警接收组支持租户内复用，不暴露数据库主键给用户。
- 告警页面中文文案恢复为正常 UTF-8 展示。
- 通知方式选项与系统现有通知渠道能力联动，Webhook 保持租户级例外。

### 2.2 不在本次范围

- 不改造告警计算引擎本身。
- 不新增告警通知模板层。
- 不兼容历史自由文本通知配置；系统当前尚未启用旧配置。

## 3. 总体方案

### 3.1 告警规则通知配置

告警规则继续使用 `alarm_rules.notify_config` 字段存储通知编排，但不再保存任意文本，而是保存结构化 JSON：

```json
{
  "version": 1,
  "channels": ["IN_APP", "EMAIL", "WEBHOOK"],
  "recipientGroupCodes": ["ARG7F3A19C2"],
  "recipientUsernames": ["zhangsan", "lisi"]
}
```

设计要点：

- `channels` 保存通知方式，不直接绑定具体渠道记录 ID。
- `recipientGroupCodes` 使用业务编码，不暴露数据库主键。
- `recipientUsernames` 使用用户名，避免把用户 ID 暴露到页面。
- 规则保存时后端会规范化 JSON，并校验通知方式枚举是否合法。

### 3.2 接收组模型

新增两张租户级表：

- `alarm_recipient_groups`
- `alarm_recipient_group_members`

其中：

- 组对外使用 `code` 作为业务唯一键。
- 成员关系表内部保存 `user_id`，页面和接口层统一使用 `username`。
- 组编码由后端自动生成，用户只维护名称、说明、成员。

### 3.3 通知方式来源

告警规则页不会让业务用户选择具体渠道实例，而是只选择“通知方式”。可选方式由通知中心实时汇总：

- 平台默认启用的 `EMAIL`、`SMS`、`PHONE`、`WECHAT`、`DINGTALK`、`IN_APP`
- 当前租户启用的 `WEBHOOK`

这样可以保持职责边界：

- 系统运维维护通道配置
- 业务侧只做告警通知编排

## 4. 前后端交互

### 4.1 新增接口

#### 系统模块

- `GET /api/v1/users/options`
  - 返回当前租户可选接收人列表
- `POST /api/v1/alarm-recipient-groups/list`
- `GET /api/v1/alarm-recipient-groups/options`
- `GET /api/v1/alarm-recipient-groups/{code}`
- `POST /api/v1/alarm-recipient-groups`
- `PUT /api/v1/alarm-recipient-groups/{code}`
- `DELETE /api/v1/alarm-recipient-groups/{code}`

#### 支撑模块

- `GET /api/v1/notifications/channel-types/available`
  - 返回当前租户可选通知方式及可用通道数量

### 4.2 页面改造

#### 告警规则页

- 抽屉表单新增“通知编排”区域
- 支持多选通知方式
- 支持多选告警接收组
- 支持多选指定接收人
- 预览区同时展示“条件摘要 + 通知摘要”

#### 告警接收组页

- 新增租户业务空间菜单 `/alarm-recipient-groups`
- 复杂表单使用抽屉，不再使用弹窗
- 成员选择全部使用下拉多选，不允许手输用户名

## 5. 数据流

### 5.1 规则维护

1. 页面加载项目、产品、设备、通知方式、接收组、接收人选项。
2. 用户在抽屉中维护规则条件和通知编排。
3. 前端将条件构造成 `conditionExpr`，将通知策略构造成 `notifyConfig`。
4. 后端对两部分 JSON 进行规范化和校验后入库。

### 5.2 后续通知执行预期

本次未改造告警触发执行链路，但运行时已经可以按以下规则解析：

1. 根据 `channels` 解析要使用的通知方式。
2. 从 `recipientGroupCodes` 找到组成员。
3. 与 `recipientUsernames` 做并集去重。
4. 再由通知中心按方式解析当前租户可用通道并发送。

## 6. 权限与菜单

- 告警接收组接口使用 `alarm:read` / `alarm:update` 权限模型。
- 告警接收组菜单挂在租户业务空间“规则告警”分组下。
- Flyway `V22__init_alarm_recipient_groups.sql` 会为已有租户菜单配置补充该入口。

## 7. 关键设计取舍

### 7.1 为什么规则里只选“方式”，不选具体通道实例

如果规则直接绑定具体渠道 ID，会让业务人员感知系统运维层的配置细节，也会把平台默认渠道和租户 Webhook 的差异带入业务页面。当前方案只选择方式，通道实例由通知中心在运行时解析，职责更清晰。

### 7.2 为什么接收组使用业务编码

用户不可见页面和文档中不应暴露数据库主键。接收组对外统一使用自动生成的 `code`，既满足业务唯一键要求，也避免用户记忆内部 ID。

### 7.3 为什么成员对外使用用户名

用户名在系统内全局唯一，适合作为规则配置和页面传值的业务键；而成员关系表内部仍保留 `user_id`，便于后续联表和性能优化。

## 8. 风险与约束

- 如果租户没有可用通知方式，规则页仍可保存纯告警规则，但不会有通知推送。
- 告警接收组成员依赖当前租户下的有效用户；用户禁用或删除后，需要重新检查接收组配置。
- 本次只完成规则编排与接收组能力，实际触发后的发送执行仍依赖后续告警触发链路接入。
