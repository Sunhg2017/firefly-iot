# 设备影子期望属性物模型预填充设计

## 背景

设备影子页面在编辑 `desired` 时原先只回填当前影子中的期望属性。对于刚接入、尚未维护完整期望属性的设备，运维人员需要手工对照产品物模型补全 JSON 结构，容易漏填字段，也不利于结构体等复杂属性的维护。

## 目标

- 在设备影子进入 `desired` 编辑态时，根据产品物模型自动生成一份初始 JSON 草稿。
- 已经存在于当前 `desired` 中的值必须保留，不能被默认值覆盖。
- 为所有带 `identifier` 的属性生成初始值，不再因为只读口径被排除在期望属性编辑区之外。
- 影子查看态仍然展示服务端真实存储的 `desired`，不展示任何前端合成结果。

## 范围

- `firefly-web/src/pages/device/DeviceShadowDrawer.tsx`
- `firefly-web/src/pages/device-shadow/DeviceShadowPage.tsx`
- `firefly-web/src/utils/deviceShadowThingModel.ts`

本次不改动影子后端接口，不改动产品物模型编辑规则。

## 方案

### 数据来源

进入影子详情后，前端并行拉取：

- 设备影子 `desired/reported/metadata`
- 影子 `delta`
- 设备所属产品的物模型

其中物模型只作为“进入编辑态时生成草稿”的参考来源，不直接影响影子展示态。为避免首次预加载遗漏或编辑器仍停留旧模型，点击“编辑”时前端还会按当前产品重新拉取一次最新物模型，再生成草稿。

### 草稿生成规则

点击“编辑”时按以下顺序生成 JSON 草稿：

1. 从物模型中筛出带 `identifier`、可用于草稿生成的属性。
2. 按属性数据类型生成默认值。
3. 将当前影子 `desired` 递归合并到默认模板上。
4. 将合并结果放入代码编辑器，进入编辑态。

### 属性筛选规则

以下属性不参与 `desired` 草稿生成：

- 缺少 `identifier` 的属性

前端不再根据 `readonly`、`accessMode` 或 `system` 口径排除属性；只要物模型属性存在 `identifier`，就会参与 `desired` 草稿生成。这样内置 `ip` 等系统固有字段、以及历史物模型里标成只读口径的自定义字段，都会先进入初始模板，避免用户在编辑期望值时还要手工补齐结构。

### 默认值映射

- `int`：优先取 `min`，否则为 `0`
- `float` / `double`：优先取 `min`，否则为 `0`，并按 `precision` 保留精度
- `bool`：`false`
- `enum`：取首个枚举候选值
- `date`：当前时间戳
- `array`：`[]`
- `struct`：递归生成对象
- `string`：空字符串，标识符为 `ip` 时默认生成 `192.168.0.10`

前端同时兼容以下两种数据类型定义口径：

- `dataType.min / dataType.max / dataType.precision`
- `dataType.specs.min / dataType.specs.max / dataType.specs.precision`

### 合并策略

前端使用递归对象合并：

- 当前 `desired` 中已有的字段优先级最高
- 物模型默认模板只补齐缺失字段
- 结构体属性支持递归补齐未设置的子字段
- 数组等非对象值仍以当前 `desired` 为准

## 关键取舍

### 为什么只在编辑态预填充

如果查看态也展示合成后的默认值，用户会误以为这些值已经被保存到服务端。为避免“看见了但没保存”的认知偏差，本次只在进入编辑态时生成草稿。

### 为什么不自动保存

物模型默认值只能作为起始模板，不能代表真实业务目标状态。仍然要求用户确认并保存，避免平台在用户无感知的情况下批量写入期望属性。

## 风险与控制

- 物模型拉取失败时，页面仍允许编辑当前 `desired`，只是无法自动补齐默认值。
- 若物模型定义不完整，草稿会只补齐已识别的属性，不阻塞人工编辑。
- 结构体递归补齐可能扩大首屏草稿内容，因此仅在用户主动点击“编辑”后执行。
