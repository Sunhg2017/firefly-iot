# 设备影子扩展抽屉设计
## 1. 背景

设备影子包含 `Desired / Reported / Delta / Metadata` 四块 JSON 数据。原先独立设备影子页新增过“全屏”能力，但设备列表中的影子抽屉没有同样入口，且原方案描述仍以全屏弹窗为主，和当前交互不一致。

随着影子字段增多，用户在卡片内查看长 JSON、对比 `Desired` 与 `Reported`、排查 `Delta` 与 `Metadata` 时，固定高度编辑器阅读成本较高，因此需要一套在两个入口都一致的扩展查看方案。

## 2. 目标

- 将独立设备影子页和设备列表中的影子抽屉统一为同一套扩展交互。
- 四张卡片都提供展开入口，但展开形态统一为右侧 `75vw` 宽度抽屉，而不是额外的全屏弹窗。
- 四张卡片在常规视图中保持统一高度与内容基线，便于横向比对。
- `Desired` 在卡片态和扩展抽屉态共用同一份编辑草稿与编辑状态，避免双轨数据。
- 保持现有影子查询、保存、清空、删除能力不变，不新增后端接口和权限分支。

## 3. 范围

本次前端改动涉及：

- `firefly-web/src/pages/device-shadow/DeviceShadowPage.tsx`
- `firefly-web/src/pages/device/DeviceShadowDrawer.tsx`
- `firefly-web/src/components/ShadowPanelFullscreenDrawer.tsx`

本次不涉及：

- 后端接口
- 设备影子存储结构
- 菜单、权限点或空间归属
- `workspace_menu_permission_catalog` 台账
- Flyway SQL

## 4. 方案

### 4.1 统一扩展态容器

新增公共组件 `ShadowPanelFullscreenDrawer`，统一承载影子卡片的扩展查看能力：

- 位置固定在右侧
- 宽度固定为 `75vw`
- 关闭时销毁内容，避免残留上一台设备的旧状态
- 复用统一标题区和额外操作区

这样独立设备影子页与设备列表影子抽屉都能使用同一套扩展容器，减少交互和实现分叉。

### 4.2 四张卡片统一提供展开入口

以下四张卡片都增加展开按钮：

- `desired`
- `reported`
- `delta`
- `metadata`

点击任意卡片的展开按钮时，只切换当前选中的面板键，再由页面底部统一渲染一个扩展抽屉，避免每张卡片分别维护自己的弹层状态。

### 4.3 Desired 草稿共享

`Desired` 既支持卡片内编辑，也支持在 `75vw` 扩展抽屉中继续编辑。为避免卡片与抽屉出现不同步草稿：

- 独立影子页复用同一份 `desiredText` 与 `editingDesired`
- 设备列表影子抽屉复用同一份 `desiredJson` 与 `editingDesired`
- 打开或关闭扩展抽屉不会复制一份新草稿
- 保存、取消、清空后由原有影子刷新流程回填最新数据
- 基于物模型预填时，系统固有属性会优先进入默认草稿，即使物模型口径标记为只读；普通只读属性仍不进入默认草稿

### 4.4 只读面板保持只读

`Reported`、`Delta`、`Metadata` 在卡片态和扩展态都只允许查看，不新增写操作。这样既保持用户认知稳定，也避免误以为这些数据可以手工修改。

### 4.5 卡片对齐策略

为保证四张卡片在两列布局中保持整齐：

- 卡片列使用拉伸布局，让同一行卡片高度一致
- 说明区预留统一最小高度，避免描述文字长短不一导致编辑器起始位置错位
- 常规视图中的四个编辑区统一高度，减少上下两行视觉跳动

### 4.6 关闭策略

为避免上下文错乱，以下场景会主动关闭扩展抽屉：

- 关闭设备列表主影子抽屉
- 删除当前设备影子
- 切换独立影子页中的设备
- 清空独立影子页当前设备选择

### 4.7 设备选择回填策略

设备选择下拉的候选项继续展示“别名 + 设备名称”的双行信息，但选中后回填为 `display（value）` 单行格式；若别名为空，则只显示设备名称，避免内容挤出输入框并影响表单对齐。

## 5. 关键交互

### 5.1 独立设备影子页

1. 用户在设备影子页选择设备并查询影子。
2. 在任一卡片点击“全屏”按钮。
3. 页面右侧打开一个宽度为 `75vw` 的扩展抽屉。
4. 若当前是 `Desired`，抽屉顶部继续提供编辑、保存、取消、清空操作。
5. 关闭抽屉后，用户回到原卡片布局继续操作。

### 5.2 设备列表中的影子抽屉

1. 用户从设备列表打开某台设备的影子详情抽屉。
2. 在任一卡片点击“全屏”按钮。
3. 在主影子抽屉之上再打开一个 `75vw` 扩展抽屉，专注查看当前面板。
4. `Desired` 继续共享编辑草稿，`Reported / Delta / Metadata` 保持只读。
5. 关闭主影子抽屉时，扩展抽屉同步关闭。

## 6. 设计取舍

- 未使用浏览器原生 Fullscreen API，而是统一改为 `Drawer`，因为后台场景下更稳定、可控，且能保留上下文层级。
- 未保留“全屏弹窗”和“抽屉展开”双轨方案，直接收口为当前的 `75vw` 抽屉实现，避免维护两套交互。
- 将扩展能力抽离为公共组件，而不是继续分别写在两个页面内，减少后续影子卡片样式和行为漂移。

## 7. 风险与验证要点

- 四张卡片在两个入口都要展示展开按钮。
- 扩展态必须是 `75vw` 抽屉，而不是 `Modal`。
- `Desired` 在卡片态与扩展态之间切换时，草稿内容和编辑状态必须一致。
- `Reported`、`Delta`、`Metadata` 只能查看，不能修改。
- 关闭主上下文或切换设备后，扩展抽屉必须自动关闭，避免展示错误设备数据。
