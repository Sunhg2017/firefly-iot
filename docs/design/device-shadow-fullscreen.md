# 设备影子卡片全屏查看设计
## 1. 背景

设备影子页面当前包含四个核心卡片：

- Desired / 期望值
- Reported / 上报值
- Delta / 差异
- Metadata / 元数据

这四块内容都以 JSON 编辑器形式呈现。随着物模型字段增多、设备影子内容变长，卡片内固定高度编辑器会让比对和排查变得吃力，尤其是 Desired 与 Reported 的长结构查看场景。

## 2. 目标

- 为设备影子页面四个卡片都提供统一的全屏入口。
- 全屏态必须复用当前卡片的数据和编辑状态，不能出现“卡片里改了一份、全屏里又是另一份”的双轨状态。
- 保持现有后端接口、影子读写逻辑和权限模型不变。

## 3. 范围

本次仅调整前端页面交互，涉及：

- `firefly-web/src/pages/device-shadow/DeviceShadowPage.tsx`

本次不涉及：

- 后端接口
- 设备影子存储结构
- 菜单、权限点或空间归属
- `workspace_menu_permission_catalog` 台账

## 4. 方案

### 4.1 统一全屏态

页面新增 `fullscreenPanelKey` 状态，取值范围为：

- `desired`
- `reported`
- `delta`
- `metadata`

四个卡片上的“全屏”按钮都只负责切换这个状态，由页面底部统一弹出一个全屏 Modal，而不是每张卡片单独维护自己的弹层。

### 4.2 Desired 编辑态与全屏态复用同一份草稿

Desired 卡片支持编辑、保存、取消、清空。为避免状态分叉，全屏弹层与卡片内编辑器复用同一个 `desiredText` 和 `editingDesired`：

- 在卡片里编辑后打开全屏，内容保持一致
- 在全屏里继续编辑或保存，回到卡片仍是同一份内容

### 4.3 只读卡片保持只读

Reported、Delta、Metadata 在普通态和全屏态都保持只读，不新增任何写操作按钮，避免误导用户。

## 5. 关键交互

1. 用户进入设备影子页面并查询设备
2. 在任一卡片点击“全屏”
3. 页面打开统一全屏弹层
4. Desired 弹层顶部显示原有编辑动作；其余卡片仅展示说明和只读内容
5. 用户关闭弹层后回到页面原位置继续操作

## 6. 风险与取舍

- 本次选择 Modal 方案而不是浏览器原生 Fullscreen API，原因是前者更稳定、无需额外权限申请，也更符合后台系统操作习惯。
- 本次没有单独抽离公共组件，因为当前全屏需求只覆盖设备影子页一处，先在页面内收口可以减少跨页面抽象成本。
- 本次没有改动菜单和权限台账，因此不需要新增 Flyway SQL。

## 7. 验证要点

- 四个卡片都能看到“全屏”按钮。
- Desired 在全屏态可继续编辑、保存、取消、清空。
- Reported、Delta、Metadata 在全屏态仅可查看。
- 切换设备或删除影子后，全屏弹层会关闭，不保留上一台设备的旧内容。
