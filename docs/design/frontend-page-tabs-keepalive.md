# 前端页面页签与保活设计说明
> 模块: `firefly-web`
> 日期: 2026-03-16
> 状态: Done

## 1. 背景

当前控制台在路由切换后会直接卸载原页面组件，导致用户返回页面时，表单输入、筛选条件和局部编辑状态全部丢失。对于设备、产品、协议等配置页面，这会明显打断连续操作体验。

同时，现有布局缺少顶部“已打开页面”承载区，用户只能依赖左侧菜单重新进入页面，无法直观看到当前工作上下文。

## 2. 目标

- 在主布局顶部提供已打开页面页签，支持快速切回已访问页面。
- 页面从当前视图切出后不卸载，重新切回时保留原有本地状态。
- 保持现有菜单、面包屑、权限校验和空间隔离逻辑不变。
- 不引入新的第三方保活依赖，直接基于现有 React Router 与布局层实现。

## 3. 方案

### 3.1 页签管理

- 在 `BasicLayout` 内维护 `openedTabs`，按访问顺序记录当前会话内打开过的业务页面路径。
- 页签仅纳入已注册的业务路由，排除 `/403` 等错误页。
- 页签标题优先使用当前菜单树中的自定义名称，保证租户空间和系统运维空间的菜单改名后，顶部页签同步生效。
- 当菜单树尚未返回时，回退到 `routes.tsx` 中的默认路由标题与图标。

### 3.2 页面保活

- 在布局层使用 `useOutlet()` 获取当前匹配页面。
- 首次进入可保活页面时，将对应 `ReactNode` 缓存到 `cachedPages[pathname]`。
- 内容区始终渲染所有已缓存页面，仅将当前激活页面设为 `display: block`，其他页面设为 `display: none`。
- 因为页面组件未卸载，所以其内部 `useState`、表单实例、抽屉临时输入等局部状态可以保留。

### 3.3 权限与空间收口

- 只有通过 `isWorkspacePathAllowed(...)` 校验的页面才允许进入页签与缓存集合。
- 当用户权限、授权菜单或所在空间变化时，会自动裁剪不可访问页签，并释放对应缓存页面，避免保留越权页面实例。
- 关闭当前页签时，自动跳转到相邻页签；若已无其他页签，则回退到当前空间首页。

## 4. 影响范围

- `firefly-web/src/layouts/BasicLayout.tsx`
- `firefly-web/src/config/routes.tsx`
- `firefly-web/src/styles/global.css`

## 5. 设计取舍

- 没有引入额外 keep-alive 组件库，避免与现有路由体系、权限校验和懒加载链路叠加新的不确定性。
- 当前保活范围限定为“当前登录会话内、已打开且未关闭的页面”。
- 本次没有跨浏览器刷新持久化页面实例。刷新后页签和组件实例会重新初始化，但不影响“切页再返回仍保留输入”的核心诉求。
- 2026-03-16 补充收口：顶部页签承载层改为轻量条形样式，避免单页签时出现整条空白白底；同时补齐 `ScheduleOutlined` 图标映射，确保菜单树中的“定时任务”图标可以正常渲染。
- 2026-03-16 二次修正：顶部页签的受控激活项改为基于“已打开页签 + 当前路由”实时派生，消除新页面首次打开时因状态异步更新而被强制回跳到首个页签的问题。
- 2026-03-16 三次修正：移除 `antd Tabs` 的内部切换行为，顶部页签改为布局层自绘按钮组，点击和关闭完全由路由状态驱动，彻底规避组件内部激活项回退导致的误跳转。
- 2026-03-16 四次修正：根因定位为 `workspaceRoutes.ts` 对重复路径只保留最后一个空间归属，导致 `/user`、`/role`、`/menu-customization` 这类双空间复用页面在系统运维空间下被误判为“跨空间访问”，随后被布局重定向回首页。现已改为按路径维护可归属空间集合，重复路径返回“多空间可用”，不再触发错误跳转。
- 2026-03-16 五次修正：头部右侧用户名触发器从 `Space + 内联样式` 收口为独立胶囊样式，避免与 header 背景叠出不自然的大白块，并统一头像、用户名和阴影表现。
- 2026-03-16 六次修正：用户名触发器进一步调整为轻量头像入口，缩小名称胶囊体积，补充下拉箭头和头像层次，保证与通知、任务图标区的视觉重量更平衡。
