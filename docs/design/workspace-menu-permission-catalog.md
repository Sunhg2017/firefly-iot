# 工作空间菜单权限目录设计

## 1. 背景

当前系统已经完成平台空间和租户空间拆分，但菜单与权限点的对应关系仍长期散落在后端硬编码里，存在三个问题：

- 运维侧无法直接核对“某个菜单属于哪个空间、对应哪些权限点”。
- 角色权限目录、租户空间授权和实际菜单配置缺少统一台账。
- 后续新增功能时，容易出现只改路由和权限判断、遗漏空间映射记录的情况。

## 2. 目标

- 新增一张正式台账表记录菜单路径、权限点和所属空间的关系。
- 以当前系统功能为基础一次性落入初始化数据。
- 后端角色权限目录改为读取台账表，而不是继续维护硬编码常量。
- 运维空间可直接查看该台账，用于排查授权范围和功能归属。

## 3. 表结构

表名：`workspace_menu_permission_catalog`

一条记录表示：

- 一个工作空间
- 一个菜单模块
- 一个权限点

核心字段：

- `workspace_scope`
  - `PLATFORM`
  - `TENANT`
- `module_key`
  - 模块唯一标识，用于把多条权限记录归并为一个功能模块
- `module_label`
  - 模块展示名称
- `menu_path`
  - 前端菜单路由
- `permission_code`
  - 权限点编码
- `permission_label`
  - 权限点说明
- `module_sort_order`
  - 模块排序
- `permission_sort_order`
  - 权限排序
- `role_catalog_visible`
  - 是否在角色授权目录展示

约束：

- 唯一键：`workspace_scope + module_key + permission_code`
- 索引：`workspace_scope + menu_path`

## 4. 设计原则

### 4.1 一空间一记录

- 如果功能只属于系统运维空间，只写一条 `PLATFORM` 记录。
- 如果功能只属于租户业务空间，只写一条 `TENANT` 记录。
- 如果功能同时属于两个空间，必须写两条记录，不能再用 `BOTH` 这类过渡口径。

### 4.2 台账与角色目录分离

- 台账会完整记录实际菜单与权限关系。
- 但不是所有记录都适合直接进入角色授权目录。
- 对于仅复用已有权限点的辅助页面，可把 `role_catalog_visible` 设为 `FALSE`，避免角色页出现重复权限分组。

### 4.3 租户空间权限仍受空间授权限制

- 台账表负责定义“如果该菜单属于租户空间，它对应哪些权限点”。
- 租户真正可分配哪些权限，仍由租户空间授权结果决定。
- 后端通过 `tenant_menu_configs` 中当前租户已授权的 `menu_path` 集合，与台账表做交集，得到当前租户可分配权限。

## 5. 后端链路调整

- `WorkspacePermissionCatalogService` 改为从 `workspace_menu_permission_catalog` 读取数据。
- `listAssignablePermissionGroupsForCurrentWorkspace`
  - 平台空间直接取 `PLATFORM` 记录。
  - 租户空间先根据已授权菜单路径过滤，再组织角色授权目录。
- `getTenantWorkspacePermissions`
  - 使用租户已授权菜单路径与台账表做交集生成最终权限集合。
- 运维空间新增只读查询接口：
  - `GET /api/v1/workspace-permission-catalog`

## 6. 前端呈现

- 运维空间不新增独立菜单，直接在现有“权限资源”页面增加“空间权限目录”标签页。
- 该页签用于查看：
  - 所属空间
  - 模块名称
  - 菜单路径
  - 权限点
  - 是否进入角色目录

## 7. 初始化数据范围

初始化数据覆盖当前系统已有菜单权限关系，包括：

- 平台空间：
  - 工作台
  - 租户管理
  - 用户管理
  - 角色管理
  - 权限资源
  - 数据字典
  - 系统设置
  - 通知渠道
  - 定时任务
  - 系统监控
  - 安全管理
  - API Key
  - 审计日志
  - 操作日志
- 租户空间：
  - 用户管理
  - 角色管理
  - 项目管理
  - 跨租户共享
  - 产品与物模型
  - 协议解析
  - 设备管理及其周边功能
  - 规则告警
  - 数据分析
  - 异步任务
  - 固件、OTA、视频

## 8. 变更规则

后续新增或调整功能时，必须同步补 Flyway SQL：

- 功能只属于一个空间：补一条记录
- 功能属于两个空间：补两条记录
- 权限点变化：同步维护对应记录
- 模块名称、菜单路径、排序变化：同步更新记录

不允许只修改前后端路由和权限判断而漏掉台账。
