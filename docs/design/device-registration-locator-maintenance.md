# 设备注册阶段维护设备标识设计

## 1. 背景

设备标识（如 `IMEI`、`ICCID`、`MAC`、`SERIAL`）原先只能在设备创建完成后，再通过设备标识弹窗单独维护。这样会带来三类问题：

- 一机一密手动创建设备后还要二次补录标识，操作链路割裂。
- 一型一密动态注册创建设备时无法一次性写入标识，协议解析和设备识别需要等待人工补录。
- 批量导入只能导入基础设备信息，导入完成后仍需逐台补齐标识。

本次改动将设备标识能力前移到“注册时”，使平台在设备首次落库时就具备完整的识别信息。

## 2. 目标与范围

### 2.1 目标

- 手动创建设备时支持维护设备标识。
- 一型一密动态注册时支持随请求一并写入设备标识。
- 批量导入设备时支持从 Excel/CSV 解析设备标识。
- 设备模拟器在 HTTP / MQTT 一型一密自动注册时支持携带设备标识。
- 不新增新的标识模型，统一复用既有 `device_locator` 体系。

### 2.2 非目标

- 不改动设备标识列表页的增删改规则。
- 不引入新的标识表或兼容双写逻辑。
- 不对历史未维护标识的设备做自动补数，历史数据由运维按需清理或补录。

## 3. 方案概览

### 3.1 统一输入模型

新增跨模块 DTO：

- `DeviceLocatorInputDTO`
  - `locatorType`
  - `locatorValue`
  - `primaryLocator`

该 DTO 作为注册阶段的统一输入结构，被以下链路复用：

- `DeviceCreateDTO`
- `DeviceBatchCreateItemDTO`
- `DeviceRegisterRequestDTO`

### 3.2 后端落库链路

后端统一复用 `DeviceLocatorService`：

1. 设备主记录先落库。
2. 标签、分组按现有逻辑继续维护。
3. 调用 `DeviceLocatorService.createBatch(deviceId, locators)` 批量写入标识。

这样能保证：

- 手动创建、批量创建、动态注册三条链路使用同一套唯一性校验。
- 主标识清理逻辑继续由 `DeviceLocatorService` 内部负责。
- 后续扩展新入口时，不需要重复实现标识落库逻辑。

## 4. 详细设计

### 4.1 手动创建设备

前端设备管理页的“新建设备”升级为抽屉，并新增“设备标识”列表输入区：

- 标识类型使用枚举选择：`IMEI / ICCID / MAC / SERIAL`
- 标识值使用文本输入
- 是否主标识使用开关

前端提交前会做轻量归一化：

- `locatorType` 转大写
- `locatorValue` 去首尾空格
- 若用户填写了标识但未指定主标识，则默认将第一条标识设为主标识

### 4.2 动态注册

动态注册请求体新增 `locators` 字段，connector 继续透传到 device 服务。

动态注册链路调整为：

1. 校验 `productKey + productSecret + deviceName`
2. 插入设备记录，初始状态仍为 `INACTIVE`
3. 若请求携带 `locators`，立即创建设备标识
4. 返回 `productKey / deviceName / deviceSecret`

这样设备在首次动态注册成功后，就能直接被协议解析器或设备识别流程按标识命中。

### 4.3 批量导入

异步导入任务在文件解析阶段新增标识列解析。

支持两类列口径：

- 通用列
  - `locatorType`
  - `locatorValue`
  - `primaryLocator`
- 快捷列
  - `imei`
  - `iccid`
  - `mac`
  - `serial` / `sn`

解析规则：

- 通用列的 `locatorType` 与 `locatorValue` 必须同时填写，否则整行报错。
- 快捷列按固定类型直接映射为设备标识。
- 若某行最终只解析出一个标识且未指定 `primaryLocator`，系统自动将其设为主标识。

### 4.4 设备模拟器自动注册

模拟器新增 `locators` 设备配置字段，并在 HTTP / MQTT 的一型一密动态注册请求中透传。

作用：

- 联调时可以直接模拟“设备首次注册即带标识”的真实链路。
- 不需要等平台注册完成后再手工进入设备管理补录标识。

## 5. 关键取舍

### 5.1 复用既有 locator 体系，而不是新增“注册标识”字段

原因：

- 平台已经有成熟的 `device_locator` 唯一性、主标识和查询模型。
- 复用可避免“注册时一套、设备详情页一套”的双轨数据结构。
- 后续设备识别逻辑继续只依赖 `device_locator`，不引入额外兼容分支。

### 5.2 导入同时支持通用列和快捷列

原因：

- 通用列适合扩展型场景。
- 快捷列适合现网最常见的 IMEI、ICCID、MAC、SN 模板，降低批量导入门槛。

### 5.3 前端不暴露数据库主键

本次交互仍然只围绕业务标识值进行维护，不向用户暴露 locator 表主键，符合仓库对“优先使用业务唯一键”的约束。

## 6. 风险与约束

- 同一产品下设备标识值仍受唯一性约束，重复导入或重复注册会失败。
- 导入文件如果同时填写多种标识，需要业务侧自行保证值真实且属于同一设备。
- 历史设备不会自动回填标识；如果历史数据影响新识别规则，需要运维侧清理或补录旧数据。
