# 租户用户管理与协议接入协同设计

## 背景

当前系统存在两个体验问题：

1. 租户业务空间缺少用户管理入口，导致租户管理员无法在自己的工作空间内直接创建成员账号。
2. 协议接入页面存在中文乱码，且协议调试、采集任务、会话绑定与产品接入、协议解析、设备管理之间缺少明确协同关系。

本次改造目标是让租户空间具备完整的用户创建入口，同时把协议接入页面统一到“产品接入 -> 协议解析 -> 设备管理”的主链路中。

## 目标

1. 租户业务空间可直接访问用户管理页面并创建用户。
2. 已配置自定义菜单的租户，也能自动看到用户管理入口。
3. 协议接入相关页面消除中文乱码。
4. 协议接入页面统一展示与产品接入、协议解析、设备管理的协同导航。
5. 采集任务、会话绑定等场景优先展示业务标识，不再以数据库主键为主视角。

## 设计方案

## 租户空间用户管理入口

### 路由分层

- `/user` 调整为共享路由，可被平台空间和租户空间共同识别。
- 租户空间在过滤路由树时，不再沿用“系统管理”分组承载 `/user`，而是动态映射为 `组织与用户` 分组。
- 为避免租户首页默认落到 `/user`，租户空间主页选择逻辑会优先跳过 `/user`，只有在租户仅拥有该路径时才使用它作为首页。

### 已有菜单配置兼容

- 已有租户可能使用自定义菜单树，且菜单树中并不存在 `/user`。
- 运行时在租户菜单配置规范化阶段自动检查 `/user` 是否缺失。
- 若当前用户具备 `user:read` 权限且菜单树中缺少 `/user`，则自动补入一个合成分组 `组织与用户`，下挂 `用户管理`。
- 这样既不要求手工重建菜单，也不影响已有协议接入分组兼容逻辑。

## 协议接入页面协同

### 页面协同原则

- 每个协议接入页统一增加“协议接入协同引导卡片”。
- 卡片直接提供跳转到 `产品接入`、`协议解析`、`设备管理` 的入口。
- 页面描述统一强调：协议调试是接入链路的一部分，业务归档仍应落在产品和设备上下文。

### 上下文绑定策略

- SNMP、Modbus 采集任务增加清晰的“关联产品 / 关联设备”说明。
- TCP/UDP 会话与端点绑定页面，使用产品名、产品 Key、设备名称展示绑定结果。
- 采集任务和绑定结果尽量展示 `productKey`、`deviceName`、`nickname`，只在系统内部保留主键传值。

### 页面文案治理

- 重写 `routes.tsx` 中的路由标签，避免导航文字继续出现乱码。
- 重写以下页面的中文文案与说明：
  - `UserList`
  - `SnmpPage`
  - `ModbusPage`
  - `WebSocketPage`
  - `TcpUdpPage`
  - `LoRaWanPage`

## 核心实现点

1. `workspaceRoutes.ts`
   - 将 `/user` 视为共享路径。
   - 租户空间将该路径重映射到 `组织与用户` 分组。
   - 调整租户默认首页选择逻辑。
2. `BasicLayout.tsx`
   - 继续保留协议接入分组兼容逻辑。
   - 增加租户菜单配置缺失 `/user` 时的运行时补位逻辑。
3. 协议接入页面
   - 新增 `ProtocolAccessGuide` 组件，统一承载协同导航。
   - 统一产品 / 设备选项的业务标识展示。
4. `routes.tsx`
   - 清理导航层面的乱码和错误标签。

## 风险与取舍

- 运行时自动补位会让租户菜单展示结果与数据库中原始菜单树不完全一致，这是有意设计，用于修复缺失入口。
- 为避免影响平台空间原有结构，租户空间的 `组织与用户` 分组只在工作区过滤和运行时菜单规范化阶段生成，不回写默认平台分组结构。
- 当前仅收口本次用户诉求相关页面，项目中其他遗留乱码页面后续可继续按模块治理。
