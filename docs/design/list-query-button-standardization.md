# 列表页查询按钮统一设计

## 背景

多个列表页只有输入框回车触发查询，没有显式“查询”按钮。用户在当前筛选条件下需要重新拉取数据时，往往只能刷新整个浏览器页面，影响日常运维和问题排查效率。

## 目标

- 为现有列表检索入口补充显式“查询”按钮。
- 保持原有筛选项、分页、接口参数和加载逻辑不变。
- 让用户在不刷新浏览器的情况下，能够对当前页面结果再次执行查询。

## 范围

本次统一调整覆盖前端已有列表检索入口的页面，包括：

- 告警规则与告警记录列表
- 告警接收组列表
- API Key 列表
- 审计日志列表
- 设备列表
- 设备标签列表
- 字典类型列表
- 固件列表
- 地理围栏列表
- 消息模板列表
- OTA 固件与任务列表
- 产品列表
- 规则引擎列表
- 安全日志列表
- 视频设备列表

## 方案

统一基于现有 `Input.Search` / `Search` 组件开启 `enterButton="查询"`：

- 保留原有 `onSearch` 处理函数，避免改动接口入参与页面状态机。
- 不新增兼容分支，不保留旧交互与新交互双轨逻辑。
- 对已经存在筛选联动的列表页，仅补查询按钮，不改其他筛选项的既有行为。

## 设计取舍

- 未额外引入公共包装组件。
  原因：本次需求是统一补入口，不涉及新的跨页状态抽象，直接复用现有搜索组件改动更小、风险更低。
- 未改为“所有筛选项都必须点查询后才生效”。
  原因：现有页面大量依赖即时筛选，整体切换交互会扩大回归面，本次只解决“缺少显式查询入口”的问题。

## 风险与验证点

- 带按钮的搜索框宽度会挤占原有筛选区空间，需要重点验证窄屏换行表现。
- 需要验证按钮点击与回车查询都能继续命中原有 `onSearch` 逻辑。
