# Firefly IoT Repository Rules

本文件为仓库级强制规则，适用于本仓库内的所有功能开发、功能修复、重构与配置变更。

## 强制规则

1. 每次功能完成后，必须先刷新相应文档，再结束本次工作。
   - 涉及功能设计、实现范围、接口、配置、运行方式、限制条件时，必须同步更新 `docs/` 下对应文档。
   - 涉及启动方式、模块说明、接入说明或全局行为变化时，必须同步更新 `README.md`、`ARCHITECTURE.md` 或对应说明文档。
   - 如果没有现成文档承接本次变更，必须在 `docs/` 下新增对应文档，而不是跳过。

2. 文档刷新完成后，必须自动提交到仓库，不能只修改不提交。
   - 提交必须包含本次功能代码与对应文档更新，保持代码和文档一致。
   - 除非用户明确禁止提交，否则功能完成后不得停留在未提交状态。

3. 对于复杂逻辑，以及涉及页面行为、前后端交互、第三方系统或第三方 SDK/协议交互的功能，必须补充必要注释。
   - 注释应说明关键流程、状态转换、边界条件、外部依赖约束或这样实现的原因，而不是重复代码表面含义。
   - 涉及页面联动、异步时序、协议编解码、设备上下行、缓存同步、消息投递、第三方回调、鉴权或兼容性处理时，默认视为需要补充注释的场景。
   - 如果代码本身已足够直白，可保持少量注释；如果逻辑不直白，则不得省略必要注释。

## 执行顺序

1. 完成功能代码。
2. 为复杂逻辑和关键交互补充必要注释。
3. 更新对应文档。
4. 进行必要验证。
5. 执行 `git add` 和 `git commit`。

## 交付要求

- 不允许在“代码已改完但文档未更新”状态下宣告完成。
- 不允许在“复杂逻辑或关键交互已落地，但缺少必要注释”状态下宣告完成。
- 不允许在“文档已更新但未提交”状态下宣告完成。
- 最终交付时，默认应处于可追溯的已提交状态。
