# Firefly IoT Repository Rules

本文件为仓库级强制规则，适用于本仓库内的所有功能开发、功能修复、重构与配置变更。

## 强制规则

1. 每次功能或模块完成后，必须先刷新相应文档，再结束本次工作。
   - 涉及功能设计、实现范围、接口、配置、运行方式、限制条件时，必须同步更新 `docs/` 下对应文档。
   - 涉及启动方式、模块说明、接入说明或全局行为变化时，必须同步更新 `README.md`、`ARCHITECTURE.md` 或对应说明文档。
   - 如果没有现成文档承接本次变更，必须在 `docs/` 下新增对应文档，而不是跳过。

2. 模块完成后，必须具备完整的三类文档，缺一不可。
   - 设计方案文档：说明背景、目标、范围、架构、流程、接口、数据结构、关键设计取舍与风险。
   - 运维说明文档：说明部署方式、配置项、依赖服务、监控告警、日志定位、常见故障、排查方法、回滚方式。
   - 使用方法说明文档：说明适用角色、页面或接口使用方式、操作步骤、输入输出示例、注意事项与限制。
   - 如果三类文档不完整，则该模块不得宣告完成。

3. 所有模块文档必须按分类放在 `docs/` 目录下，不能继续散放。
   - 设计方案放在 `docs/design/`
   - 运维说明放在 `docs/operations/`
   - 使用方法说明放在 `docs/user-guide/`
   - 如需补充阶段进度、迁移记录或专题材料，可放在其他分类目录，但模块交付所需的三类核心文档必须优先落在上述目录。

4. 文档刷新完成后，必须自动提交到仓库，不能只修改不提交。
   - 提交必须包含本次功能代码与对应文档更新，保持代码和文档一致。
   - 除非用户明确禁止提交，否则功能完成后不得停留在未提交状态。

5. 每次提交完成后，必须继续推送到远端仓库，不能只保留本地提交。
   - 默认应执行与当前分支对应的远端推送，确保远端仓库与本地最新交付保持一致。
   - 除非用户明确禁止推送、远端不可用或存在明确的发布风险，否则不得停留在“本地已提交、远端未更新”状态。
   - 如果推送失败，必须在交付说明中明确失败原因、当前分支、当前提交号以及阻塞点。

6. 对于复杂逻辑，以及涉及页面行为、前后端交互、第三方系统或第三方 SDK/协议交互的功能，必须补充必要注释。
   - 注释应说明关键流程、状态转换、边界条件、外部依赖约束或这样实现的原因，而不是重复代码表面含义。
   - 涉及页面联动、异步时序、协议编解码、设备上下行、缓存同步、消息投递、第三方回调、鉴权或兼容性处理时，默认视为需要补充注释的场景。
   - 如果代码本身已足够直白，可保持少量注释；如果逻辑不直白，则不得省略必要注释。

## 执行顺序

1. 完成功能代码。
2. 为复杂逻辑和关键交互补充必要注释。
3. 补齐并更新模块所需的设计方案、运维说明、使用方法说明文档。
4. 将文档按分类归档到 `docs/` 对应目录。
5. 进行必要验证。
6. 执行 `git add` 和 `git commit`。
7. 执行 `git push`，将当前交付推送到远端仓库。

## 交付要求

- 不允许在“代码已改完但文档未更新”状态下宣告完成。
- 不允许在“模块已完成但设计、运维、使用文档不完整”状态下宣告完成。
- 不允许在“文档未按分类放入 `docs/` 目录”状态下宣告完成。
- 不允许在“复杂逻辑或关键交互已落地，但缺少必要注释”状态下宣告完成。
- 不允许在“文档已更新但未提交”状态下宣告完成。
- 不允许在“本地已提交但未推送远端”状态下宣告完成。
- 最终交付时，默认应处于可追溯、已提交且已推送的状态。
