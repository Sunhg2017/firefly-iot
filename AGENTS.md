# 增补强制规则（2026-03-14）

- 没有找到问题根因、证据不足，或存在关键前提/影响范围不明确时，禁止凭猜测直接修改代码、配置、SQL、页面或接口；必须先向用户同步当前不确定点，并在得到确认后再继续。
- 系统未正式使用时，禁止为了兼容旧实现、旧菜单、旧权限口径或旧数据保留双轨、兜底或灰度过渡逻辑。
- 功能重构应直接收口到当前最佳实现，旧入口、旧接口、旧配置项、旧页面和旧权限模型应一并删除，而不是继续保留备用分支。
- 如果担心历史数据影响新实现，应在交付说明和运维文档中明确提醒清理数据库旧数据，而不是在代码中继续叠加兼容分支。
- 菜单、权限点或所属空间发生变动时，必须同步通过 Flyway SQL 维护系统菜单管理基础数据，至少包括 `workspace_menu_catalog` 与 `workspace_menu_permission_catalog`；仅属于一个空间的功能写一条菜单记录，共享于系统运维空间和租户空间的功能分别写对应空间记录，禁止只改前后端代码而不更新台账。
- 新增菜单时必须同步补齐菜单 icon；`workspace_menu_catalog.icon`、前端菜单配置以及 `firefly-web/src/config/iconMap.tsx` 中的图标映射必须同时落地，禁止新增无 icon 菜单或只写图标名但前端无法渲染。

# Firefly IoT Repository Rules

## 页面文案强制规则

- 前端页面中的说明文案只允许保留对当前操作直接有帮助的内容，例如操作步骤、前置条件、结果影响、限制条件和失败处理提示。
- 禁止在页面说明、页头描述、说明卡片、提示文案中加入设计思路、视觉风格说明、理念表达、模块背景介绍、能力宣传或其他不直接服务当前操作的内容。
- 如果某段文案不影响用户完成当前页面操作，应优先删除，而不是保留为“补充说明”。
- 页面需要介绍设计取舍、改版思路、实现背景或专题说明时，必须写入 `docs/` 文档，不得继续放在页面里。

本文件为仓库级强制规则，适用于本仓库内的所有功能开发、功能修复、重构与配置变更。

## 强制规则

1. 每次功能或模块完成后，必须先刷新相应文档，再结束本次工作。
   - 涉及功能设计、实现范围、接口、配置、运行方式、限制条件时，必须同步更新 `docs/` 下对应文档。
   - 涉及启动方式、模块说明、接入说明或全局行为变化时，必须同步更新 `README.md`、`ARCHITECTURE.md` 或对应说明文档。
   - 如果没有现成文档承接本次变更，必须在 `docs/` 下新增对应文档，而不是跳过。

2. 模块完成后，必须具备完整的三类文档，缺一不可。
   - 设计方案文档：说明背景、目标、范围、架构、流程、接口、数据结构、关键设计取舍与风险。
   - 运维说明文档：说明部署方式、配置项、依赖服务、监控告警、日志定位、常见故障、排查方法、回滚方式。
   - 使用方法说明文档：说明适用角色、页面或接口使用方式、操作步骤、输入输出示例、注意事项与限制。
   - 如果三类文档不完整，则该模块不得宣告完成。

3. 所有模块文档必须按分类放在 `docs/` 目录下，不能继续散放。
   - 设计方案放在 `docs/design/`
   - 运维说明放在 `docs/operations/`
   - 使用方法说明放在 `docs/user-guide/`
   - 如需补充阶段进度、迁移记录或专题材料，可放在其他分类目录，但模块交付所需的三类核心文档必须优先落在上述目录。

4. 文档刷新完成后，必须自动提交到仓库，不能只修改不提交。
   - 提交必须包含本次功能代码与对应文档更新，保持代码和文档一致。
   - 除非用户明确禁止提交，否则功能完成后不得停留在未提交状态。

5. 每次提交完成后，必须继续推送到远端仓库，不能只保留本地提交。
   - 默认应执行与当前分支对应的远端推送，确保远端仓库与本地最新交付保持一致。
   - 除非用户明确禁止推送、远端不可用或存在明确的发布风险，否则不得停留在“本地已提交、远端未更新”状态。
   - 如果推送失败，必须在交付说明中明确失败原因、当前分支、当前提交号以及阻塞点。

6. 对于复杂逻辑，以及涉及页面行为、前后端交互、第三方系统或第三方 SDK/协议交互的功能，必须补充必要注释。
   - 注释应说明关键流程、状态转换、边界条件、外部依赖约束或这样实现的原因，而不是重复代码表面含义。
   - 涉及页面联动、异步时序、协议编解码、设备上下行、缓存同步、消息投递、第三方回调、鉴权或兼容性处理时，默认视为需要补充注释的场景。
   - 如果代码本身已足够直白，可保持少量注释；如果逻辑不直白，则不得省略必要注释。

7. 前端页面设计应尽量减少用户手工输入，凡是可以稳定枚举、查询或联动选择的字段，优先使用选择型控件，不允许默认让用户手填。
   - 能以下拉选择解决的字段，优先使用下拉选择。
   - 能用单选、多选、级联选择、自动完成、日期时间选择、开关或步进器表达的字段，不应退化为普通文本输入框。
   - 产品、设备、租户、协议、状态、类型、方向、版本、策略、消息类型等存在稳定候选集的字段，默认应提供可选项。
   - 上下游字段存在依赖关系时，应优先做联动选择、默认值或预填充，避免用户记忆和手输 ID、编码或固定枚举值。
   - 只有在候选值无法预先枚举、无法可靠查询或业务本质上就是自由输入时，才允许使用手工输入框。

8. 前端表单容器必须按复杂度选择交互形式，复杂表单强制使用抽屉，简单表单才允许使用弹窗。
   - 复杂表单包括但不限于：字段较多、分区较多、存在明显配置步骤、包含 `Form.List`、包含调试/预览/脚本/JSON 配置、包含联动选择或长文本配置的页面。
   - 简单表单包括但不限于：少量基础字段录入、确认操作、只读详情、小范围状态调整。
   - 对于创建、编辑、调试、注册、配置类页面，如果用户需要边看列表边持续填写、比对或多次修改，默认视为复杂表单，应使用抽屉而不是弹窗。
   - 评审和自检时，必须检查复杂表单是否仍误用弹窗；如已落地为弹窗，必须在交付前改为抽屉。

9. 系统数据库主键尽量不要暴露给前端用户，凡是存在业务唯一键的场景，优先使用业务唯一键而不是数据库主键。
   - 前端页面、接口出参与文档示例中，默认不应直接展示数据库自增 ID、内部主键 ID 或仅用于系统内部关联的标识。
   - 产品、租户、设备、规则、项目、字典、模板等对象，如存在 `code`、`productKey`、`tenantCode`、`deviceName`、`key`、`slug` 等业务唯一键，应优先使用这些字段作为展示、选择、搜索、复制和联动依据。
   - 如果前后端交互或底层存储仍必须依赖主键，前端应尽量将主键隐藏在内部传值层，不应把主键作为用户主视角信息。
   - 新增页面字段、调试参数、筛选条件、弹窗标题、表格列、配置示例和文档说明时，必须先判断能否改为业务唯一键口径；只有确实不存在稳定业务唯一键时，才允许暴露主键。
   - 涉及默认值、模板、预填配置或导入导出示例时，应优先补齐业务唯一键，避免要求用户记忆或手填数据库主键。

10. 数据导入导出必须走异步任务中心，禁止前端解析文件。
   - 前端不允许直接解析导入文件（如 Excel、CSV），文件解析必须由后端异步任务处理。
   - 导入流程：前端将文件上传到 MinIO，获取 `fileKey`，然后调用后端接口注册异步导入任务，后端根据 `fileKey` 读取文件并异步处理。
   - 导出流程：前端调用后端接口注册异步导出任务，后端异步生成文件并存储到 MinIO，完成后通知前端下载链接。
   - 导入导出任务必须支持进度查询、状态跟踪（待处理、处理中、已完成、失败）和错误日志记录。
   - 禁止同步阻塞式导入导出，大文件或大批量数据必须走异步任务，避免接口超时或前端卡死。

11. 禁止在 Java 代码里使用 `@Select` 直接写 SQL，所有 SQL 必须在 `mapper.xml` 中维护。
   - 不允许在 Mapper 接口、Service、Repository 或其他 Java 类中通过 `@Select` 注解内联 SQL。
   - 新增查询、改造旧查询或重构 SQL 时，必须将 SQL 统一放入对应的 `mapper.xml`，由 Mapper 接口只保留方法声明。
   - 涉及复杂查询、动态条件、联表、分页、聚合、排序、导入导出查询时，默认都必须使用 `mapper.xml`，不得回退为注解 SQL。
   - 评审和自检时，必须检查是否新增了 `@Select` 注解 SQL；一经发现，必须在交付前改回 `mapper.xml`。

12. 新增、删除、迁移或改名菜单时，必须同步通过 Flyway SQL 增减系统菜单管理基础数据。
   - 新增菜单时，至少同步维护 `workspace_menu_catalog`（含 `icon` 字段）、`workspace_menu_permission_catalog` 以及相关 `permission_resources`；如果前端运行时依赖图标映射，必须同步补齐前端图标映射。
   - 删除或替换旧菜单时，必须同步清理旧菜单对应的 `workspace_menu_catalog`、`workspace_menu_permission_catalog`，以及相关 `workspace_menu_customizations`、`tenant_menu_configs` 等脏数据。
   - 只改前端路由、后端接口或权限常量，而不补系统菜单管理基础数据 SQL，视为不合格交付。

## 执行顺序

1. 完成功能代码。
2. 为复杂逻辑和关键交互补充必要注释。
3. 前端页面涉及表单或筛选时，优先收敛为选择型交互，减少手工输入。
4. 前端页面涉及创建、编辑、调试、注册、配置类表单时，必须判断是否属于复杂表单；复杂表单必须改为抽屉，简单表单才可保留弹窗。
5. 检查用户可见交互是否错误暴露了数据库主键，能改为业务唯一键的必须收口为业务唯一键。
6. 补齐并更新模块所需的设计方案、运维说明、使用方法说明文档。
7. 将文档按分类归档到 `docs/` 对应目录。
8. 若涉及菜单、权限点或所属空间变动，检查 Flyway SQL 是否同步维护系统菜单管理基础数据。
9. 检查 Java 持久层是否存在 `@Select` 注解 SQL，发现后必须收敛到 `mapper.xml`。
10. 进行必要验证。
11. 执行 `git add` 和 `git commit`。
12. 执行 `git push`，将当前交付推送到远端仓库。

## 交付要求

- 不允许在“代码已改完但文档未更新”状态下宣告完成。
- 不允许在“模块已完成但设计、运维、使用文档不完整”状态下宣告完成。
- 不允许在“文档未按分类放入 `docs/` 目录”状态下宣告完成。
- 不允许在“复杂逻辑或关键交互已落地，但缺少必要注释”状态下宣告完成。
- 不允许在“前端可选字段仍要求用户手工输入”状态下宣告完成。
- 不允许在“复杂表单仍使用弹窗，而未按规则改为抽屉”状态下宣告完成。
- 不允许在“用户可见页面、接口示例或文档说明仍直接暴露数据库主键，而本可以使用业务唯一键”状态下宣告完成。
- 不允许在“导入导出未走异步任务中心或前端直接解析文件”状态下宣告完成。
- 不允许在“Java 代码中仍使用 `@Select` 直接写 SQL，且未收敛到 `mapper.xml`”状态下宣告完成。
- 不允许在“新增、删除或迁移菜单，但未通过 Flyway SQL 同步系统菜单管理基础数据”状态下宣告完成。
- 不允许在“新增菜单但未补齐菜单 icon、前端图标映射或对应基础数据”状态下宣告完成。
- 不允许在“问题根因未明、关键前提不清或证据不足”的情况下，基于猜测直接修改后宣告完成。
- 不允许在“文档已更新但未提交”状态下宣告完成。
- 不允许在“本地已提交但未推送远端”状态下宣告完成。
- 最终交付时，默认应处于可追溯、已提交且已推送的状态。
